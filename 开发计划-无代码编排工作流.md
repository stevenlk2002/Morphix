# 开发计划 - 无代码编排工作流

> **版本**：v1.0 | **日期**：2026-07-11 | **状态**：初稿

---

## 一、计划目标

本计划用于指导“多项目私域运营平台中的多渠道无代码编排子系统”从 0 到 1 建设，按照里程碑方式拆分目标、范围、产出、职责和风险，确保产品、设计、前后端、测试和运维能够协同推进。

---

## 二、总体策略

### 2.1 建设原则

1. **先主链路，后治理增强**：优先打通“编排 -> 调试 -> 发布 -> 运行”主链路。
2. **先通用能力，后渠道扩展**：先抽象节点模型和执行引擎，再做渠道适配扩展。
3. **先单流程稳定，后模板复用**：先保证核心工作流的可用性和可观测性，再做子流程、模板、嵌入式复用。
4. **先平台内闭环，后外部生态**：首版不追求开放插件生态，优先把知识库、素材库、标签、托管能力打通。

### 2.2 里程碑划分

- **M1：MVP 基础可用版**
- **M2：Beta 业务可运营版**
- **M3：GA 正式生产版**

---

## 三、里程碑规划

## 3.1 M1：MVP 基础可用版

### 目标
完成从工作流编辑、保存、调试到运行的最小闭环，支撑基础对话机器人与知识机器人搭建。

### 范围

1. 编排画布基础能力
2. 节点拖拽、连线、删除、视图管理
3. 节点类型：
   - 用户输入
   - AI 对话
   - 消息输出
   - 对话记录获取
   - 知识库搜索
   - 多重判断器
   - 文本包装
4. 全局变量与局部变量
5. 对话测试与逐节点调试
6. 草稿保存与基础发布
7. 基础运行引擎
8. 基础知识库和素材库接入

### 产出

- 前端编排工作台 V1
- 工作流定义服务 V1
- 工作流执行引擎 V1
- 调试服务 V1
- 基础发布服务 V1
- 6~8 个基础节点执行器

### 验收标准

- 可搭建“复读机”“有记忆机器人”“单领域知识机器人”
- 必填校验、变量引用、连线校验有效
- 对话测试可查看节点执行详情
- 已发布工作流可被机器人加载并运行

### 建议周期

- 4~6 周

---

## 3.2 M2：Beta 业务可运营版

### 目标
增强工作流的复用性、知识能力和治理能力，使其可被项目团队稳定使用。

### 范围

1. 新增节点类型：
   - JSON 节点
   - 正则节点
   - 脚本运行
   - HTTP 请求
   - 时间控制
   - 文件/图片/视频/语音/链接卡片输出
   - 高级知识库搜索
2. 子流程调用与智能体嵌入
3. 高级知识库导入与字段配置
4. 版本管理、差异查看、回滚
5. 审计日志
6. 角色权限控制
7. 机器人参数联动（欢迎语、多模态识别、打断处理等）
8. 运行日志和基础监控指标

### 产出

- 节点协议与注册中心 V1
- 子流程服务 V1
- 高级知识库服务 V1
- 版本治理与回滚能力 V1
- 权限治理服务 V1
- 审计日志服务 V1

### 验收标准

- 可搭建多分支、多知识源、多输出类型的机器人流程
- 子流程可独立维护并被主流程复用
- 发布版本可回滚，关键操作可审计
- 项目内不同角色权限边界清晰

### 建议周期

- 5~7 周

---

## 3.3 M3：GA 正式生产版

### 目标
形成多渠道可扩展、治理完善、监控完整的正式生产系统。

### 范围

1. 特殊渠道节点完善：
   - 获取/设置企业微信标签
   - 获取/设置 Morphix 标签
   - 拉群
   - 取消 AI 托管
2. 渠道适配器架构完善，支持多渠道统一能力抽象
3. 运行时性能优化
4. 节点失败重试、降级与告警
5. 调试回放与问题定位工具增强
6. 模板复制、跨项目迁移能力
7. 更完整的监控看板（成功率、耗时、知识命中率、渠道失败率）
8. 发布审批与高危动作保护（可选）

### 产出

- 渠道适配层 V1
- 运行监控与告警体系 V1
- 模板与迁移能力 V1
- 稳定性与性能优化方案 V1

### 验收标准

- 多项目、多渠道下稳定运行
- 工作流发布、回滚、运行和审计闭环完整
- 核心链路达到性能指标要求
- 线上故障可通过日志和回放快速定位

### 建议周期

- 6~8 周

## 七、风险与对策

| 风险 | 描述 | 对策 |
|------|------|------|
| 节点模型不稳定 | 前期节点协议变化大，影响前后端联调 | 先冻结节点协议 v1，再逐步扩展 |
| 执行引擎复杂度高 | 节点依赖、并发、异常链路复杂 | M1 只支持 DAG 主链路，逐步增强 |
| 知识检索效果波动 | 不同知识结构下检索策略差异大 | 明确普通/高级知识库边界并可切换模式 |
| 渠道差异过大 | 多渠道动作不一致 | 通过适配器层做统一抽象 |
| 调试与生产不一致 | 草稿态与发布态行为差异 | 调试时明确运行对象（草稿/版本）并写入日志 |
| 权限治理过晚介入 | 后补权限会导致返工 | M2 必须纳入权限和审计设计 |
| 性能瓶颈 | 中大型流程和高并发会话下响应变慢 | 引擎异步化、缓存、指标监控前置设计 |

## 八、关键里程碑检查点

### M1 结束检查点
- 是否已能完成基础机器人编排闭环
- 是否能支持最小知识问答流程
- 调试详情是否足以定位节点问题

### M2 结束检查点
- 是否已具备正式给业务团队使用的治理能力
- 子流程和高级知识库是否稳定
- 发布/回滚/审计链路是否闭环

### M3 结束检查点
- 是否支持多渠道正式生产运行
- 是否有足够的监控和告警手段
- 是否形成稳定上线流程和运维手册

## 十、上线建议

### 10.1 灰度策略

1. 先在单项目、单机器人、单渠道灰度
2. 再扩展到多机器人、多知识库场景
3. 最后开放多渠道和高风险动作节点

### 10.2 发布建议

- 发布必须填写变更说明
- 高风险动作节点建议默认关闭或需审批
- 对调试流量和生产流量使用独立标记

### 10.3 运行保障

- 建立工作流版本回滚预案
- 建立关键节点失败告警
- 建立 AI 超时和知识库空结果兜底策略
