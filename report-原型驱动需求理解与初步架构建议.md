# Morphix 原型驱动需求理解与初步架构建议

## 1. 我先给结论

这不是一个“聊天机器人后台”，而是一个 **面向多项目、多渠道、多账号、多Agent协同的私域运营执行平台**。

并且从你给的原型来看，真正的产品核心不是单点 AI 对话，而是下面这条闭环：

**项目配置 -> 机器人定义 -> 无代码工作流编排 -> 渠道账号托管 -> 客户/群会话处理 -> 标签沉淀 -> SOP运营任务 -> 数据监控 -> 持续优化**

如果只把它理解成“做一个工作流编辑器”，方向会偏；
如果只把它理解成“做一个多Agent系统”，也会偏。

**原型体现出的真正核心，是“用可配置机器人 + 可编排工作流 + 渠道执行网络”，把销售型私域运营流程产品化。**

---

## 2. 需求理解的优先级

本次理解我采用如下优先级：

1. **原型 `prototype/index.html` 作为需求主锚点**
2. PRD 用来补业务目标、场景边界和非功能要求
3. 编排手册/节点图/默认变量文档，用来补齐工作流运行模型
4. 多Agent 协同 PDF，以 PRD 中已整合出的协同链路作为主要依据

这意味着：
- **界面信息架构、用户操作路径、模块组织，以原型为准**
- **业务愿景、规模目标、Agent 职责分工，以 PRD 为准**
- **节点协议、端口类型、变量体系，以编排手册和原型定义为准**

#### C. 客户生命周期

## 4. 产品本质：三层业务模型

我建议把这个产品理解成三层：

### 第一层：业务控制层（Control Plane）
给运营/实施/项目负责人用。

包括：
- 项目管理
- 机器人管理
- 知识库/素材库
- 无代码编排
- SOP任务配置
- 组织权限
- 数据面板

这一层决定“**该怎么做**”。

### 第二层：会话执行层（Execution Plane）
给系统实时跑业务。

包括：
- 渠道消息接入
- 会话路由
- 多Agent执行
- 工作流调度
- 知识检索
- 消息生成
- 标签更新
- 运营任务触发

这一层决定“**现在做什么**”。

### 第三层：设备边缘层（Edge Device Plane）
给手机端 App 用。

包括：
- 登录具体渠道账号
- 联系人/群聊/动态采集
- 消息收发
- 指令执行
- 回执上报
- 风险控制

这一层决定“**由谁去执行渠道动作**”。

这个分层非常重要，因为你第 6 点已经明确：

> 手机端不是附属页面，而是单个渠道账号的真实执行站点。

所以手机端不是普通客户端，而是 **边缘执行节点**。

## 5. 你这个系统的核心业务目标，我理解成 6 件事

### 5.1 一个平台承载多个项目
每个项目独立：
- 机器人配置
- 知识库
- 工作流
- 客户数据
- 渠道账号
- 运营SOP
- 指标分析

也就是说，**Project 是一级租户，不是简单标签字段。**

### 5.2 一个项目可以挂很多渠道账号
包括：
- 微信个人号
- 企业微信
- WhatsApp 个人号
- WhatsApp Business

但从你的现阶段重点看，**微信 / 企业微信 + 手机侧执行** 是近期核心。

### 5.3 一个渠道账号可绑定一个预设机器人
这里不是“账号 = 机器人”这么死。
更准确地说，是：

- 账号有默认托管机器人
- 会话可切换托管机器人
- 任务执行后可指定后续托管机器人
- 群聊和单聊可有不同托管策略

所以真实模型应该是：

**账号、会话、机器人之间是可配置映射关系，不是一对一硬绑定。**

### 5.4 机器人能力来自模板继承 + 工作流重编排
原型已经体现了两条路：
- 选择模板创建机器人
- 从编排创建机器人

这说明你的机器人体系不是纯 Prompt，而是：

**Bot = 业务元数据 + 能力配置 + Workflow版本引用 + 知识资产绑定**

### 5.5 多Agent 不只是“多个模型”，而是分工协作
从 PRD 中整合的链路看，至少包括：
- 意图分析/需求分析
- 知识检索
- 闲聊处理
- 阶段推进
- 表达控制
- 后置控制
- 标签提取
- 主动触达决策

所以产品内的“Agent”应被定义为：

**可被编排的专业能力单元，而不是一个独立聊天人格。**

### 5.6 最终目标是替代人工销售运营中的大部分机械工作
不是 100% 替代人，而是：
- 高并发接待
- 低风险触达
- 标签沉淀
- SOP 自动化
- 高意向转人工
- 可追踪、可回放、可优化

换句话说，平台目标不是“最聪明”，而是“**最稳定地完成转化型运营工作**”。

## 6. 无代码编排部分，我理解到的运行模型

### 6.1 节点模型已经很清楚
原型和规范文档基本一致，节点分成：
- 获取内容
- 输出
- 工具
- 流程控制
- 逻辑处理
- 复合
- 特殊渠道

### 6.2 七种输入输出点类型
结合原型和你给的节点图，当前端口类型可抽象为：
- 字符串
- 数字
- 布尔
- 聊天记录
- 引用内容（知识结果）
- 任意类型
- 属性

这说明你不是做“弱表单编排”，而是在做 **强类型数据流编排**。

### 6.3 默认变量体系已经具备业务运行味道
比如：
- `userChatInput`
- `AIAnalyzeChatInput`
- `chatId`
- `messageType`
- `innerUserId`
- `externalUserId`
- `channelType`
- `roomTopic`
- `nowTime`
- `interruptOldOnNew`

这说明工作流运行时上下文不是通用自动化上下文，而是 **面向“渠道会话托管”场景定制的运行时上下文模型**。

### 6.4 编排引擎的真正难点不是画布，而是运行时
难点在：
- 变量解析
- 节点类型校验
- 分支调度
- 异常策略
- 多轮上下文
- 子流程版本引用
- 调试回放
- 渠道动作幂等

所以画布前端是门面，**真正的核心壁垒在工作流运行时。**

## 7. 多Agent 应该怎么落到产品里

我不建议把多Agent设计成“几个 Agent 互相自由聊天”的开放式结构，那样很炫，但业务不可控。

更适合你这个系统的是：

## 7.1 更合适的结构：总控编排层 + 三类 Agent + 可选 Supervisor Agent

先说结论：这里确实需要“总控能力”，但我不建议把它一开始就设计成一个每轮必经的“总控Agent”。

更适合你这个系统的是：
- 由 **Workflow Runtime / Session Orchestrator** 承担确定性主调度
- 由 **Policy Router** 承担路由、风控、模型与成本策略
- 由三类 Agent 负责具体能力执行
- 在复杂场景下再引入 **Supervisor Agent** 做策略辅助

这样做的好处是：
- 可回放
- 可审计
- 成本可控
- 不容易出现“AI 上帝进程”单点失控

### A. 总控编排层（不是 Agent，而是系统运行时）
职责：
- 会话入口路由
- Bot / WorkflowVersion 选择
- 节点调度与上下文推进
- 中断恢复
- 超时、重试、幂等
- 人工接管 / 交还
- 设备执行回执处理

其中建议再拆成两个子角色：
- **Session Orchestrator**：负责会话状态机和工作流推进
- **Policy Router**：负责策略路由、风控、优先级、模型选择、成本控制

### B. 在线会话 Agent
直接参与单聊/群聊实时处理：
- 需求分析 Agent
- 知识问答 Agent
- 销售推进 Agent
- 表达控制 Agent
- 风险兜底 Agent

### C. 离线分析 Agent
异步执行，不阻塞主回复：
- 客户偏好识别 Agent
- 客户信息提取 Agent
- 黑名单/剔除识别 Agent
- 群活跃/进群状态 Agent
- 朋友圈内容分析 Agent

### D. 运营编排 Agent
不直接回复客户，而是驱动运营任务：
- 目标客户筛选 Agent
- 触达时机判断 Agent
- SOP内容装配 Agent
- 运营结果评估 Agent

### E. Supervisor Agent（可选）
不建议让它成为每轮必经链路，更适合在高复杂度场景下参与：
- 复杂目标拆解
- 多方案策略比较
- 异常恢复建议
- 高价值客户推进策略
- 跨 Agent 协调建议

## 7.2 Agent 在系统里的正确位置

Agent 不应该成为唯一总控者，也不应该成为顶级业务对象；
对业务来说，顶级对象仍然应该是：
- 项目
- 机器人
- 工作流
- 会话
- 任务

Agent 更适合作为：

- 节点执行器的一种
- 或工作流里的能力单元
- 或运行时可路由的策略模块
- 或在复杂场景下参与决策的辅助角色

也就是：

**Bot 面向业务交付，Agent 面向能力编排；总控主要由编排运行时承担，Supervisor Agent 只在必要时参与。**

这个边界一定要分清，不然后面产品会越来越乱。

## 9. 架构选型：我建议不要一步上纯微服务

## 方案 A：模块化单体 + 独立执行组件（推荐）

### 结构
- 一个主业务后端（项目/机器人/工作流/客户/运营/权限）
- 一个工作流执行服务（可先同仓，后独立部署）
- 一个设备接入网关
- 一个异步任务队列
- Redis + PostgreSQL + 对象存储 + 向量检索

### 优点
- 初期成本低
- 研发效率高
- 事务一致性简单
- 更适合需求快速变化期

### 代价
- 后期边界若没提前收好，会有拆分痛苦
- 热点模块容易互相影响

## 方案 B：从第一天就全微服务

### 结构
- Project 服务
- Bot 服务
- Workflow 服务
- Conversation 服务
- Device Gateway
- Campaign 服务
- Analytics 服务
- Knowledge 服务

### 优点
- 理论上边界最清晰
- 独立扩缩容方便

### 代价
- 初期研发/联调/运维成本高
- 业务还没稳定时，接口会反复改
- 小团队很容易被服务治理拖死

## 我的结论

**建议采用 A：模块化单体 + 独立接入/执行/队列组件 的混合架构。**

这不是保守，而是可逆性最高。

你当前初期目标是：
- 1000 渠道账号
- 每账号 1000+ 联系人/群
- 同时支持托管与运营任务

这个规模已经不算“小玩具”，但又没大到必须全微服务。

**最适合的路径是：控制面先模块化单体，执行面从一开始就解耦。**

## 10. 推荐的总体技术架构

## 10.1 控制面（Control Plane）
职责：配置和治理。

建议包含：
- Web 前端（React）
- 主 API 服务
- 权限/审计
- 项目/Bot/Workflow/Knowledge/SOP 管理

特点：
- 读写多
- 强事务
- 管理后台为主
- 更适合关系型数据库

## 10.2 执行面（Execution Plane）
职责：实时会话和任务执行。

建议包含：
- 会话路由器
- 工作流运行时
- Agent 执行器
- 知识检索服务
- 消息发送策略器
- 异步标签/分析任务

特点：
- 高并发
- 强异步
- 事件驱动
- 可水平扩展

## 10.3 边缘设备面（Edge Plane）
职责：真实渠道动作执行。

建议包含：
- Android App（优先）
- 设备心跳
- 账号状态
- 消息监听/上传
- 发送指令执行
- 联系人/群/动态同步
- 风险策略开关

特点：
- 连接不稳定
- 网络环境复杂
- 必须离线容忍
- 必须支持回执重传

## 11. 我建议的关键链路设计

## 11.1 实时消息链路

**手机/官方渠道 -> 接入网关 -> 统一事件模型 -> 会话路由 -> 工作流执行 -> 消息策略 -> 下发渠道动作 -> 设备执行 -> 回执 -> 会话落库**

关键点：
- 接入先标准化，不要让上层直接感知渠道差异
- “会话事件”和“设备事件”要分开建模
- 回复生成和发送执行分离

## 11.2 主动运营链路

**定时器/运营任务 -> 人群筛选 -> 触达策略判断 -> 工作流/模板渲染 -> 下发执行 -> 回执统计 -> 效果分析**

关键点：
- 运营任务不是即时聊天逻辑的附属品
- 它应该是独立调度系统

## 11.3 标签异步链路

**会话结束/消息到达 -> 投递异步分析任务 -> 多Agent抽取/判断 -> 标签更新 -> 客户画像更新 -> 触发后续SOP**

关键点：
- 标签分析不要阻塞主回复
- 标签更新要可追踪、可回滚、可重算

## 12. 手机端 App 的架构定位

这个点我特别强调一下。

原型里有“移动端会话托管管理”的小程序形态，但那是 **运营人员的移动管理界面**。

而你第 6 点要求的，其实是另一套东西：

## 12.1 设备侧执行 App
它是：
- 安装在 1000 台手机上的边缘执行器
- 绑定单个或少量渠道账号
- 负责采集/发送/同步/回执
- 接受服务端指令

所以它不是“小程序”，而应该是 **独立移动端执行 App**。

## 12.2 两类移动端不要混

### A. 运营移动端
给管理者用：
- 看会话
- 筛选托管
- 批量分配机器人
- 紧急接管

原型里已经体现了这部分。

### B. 设备执行端
给设备用：
- 登录渠道账号
- 维持在线
- 监听消息
- 执行动作
- 上传联系人/群/动态

这两类移动端的目标完全不同，建议从架构、代码库、权限、部署全部分开。

## 13. 1000 账号初期目标下，我建议的容量思路

## 13.1 先按初期目标设计
初期设计目标：
- 1000 渠道账号在线
- 每账号 1000+ 联系人/群
- 并发会话 1 万级以内可控
- 主回复 P99 < 3 秒

这时重点不是极限吞吐，而是：
- 接入稳定
- 队列削峰
- 可观测性
- 风险隔离
- 成本可控

## 13.2 成本友好的初期组合
建议：
- **PostgreSQL**：核心交易/配置/客户/会话元数据
- **Redis**：会话热状态、去重、限流、短期上下文缓存
- **MQ**（RabbitMQ/Kafka 二选一）：消息事件与异步任务解耦
- **对象存储**：图片/语音/视频/文件
- **pgvector 或独立向量库**：知识检索

## 13.3 为什么现在不必按 5 万账号一次性设计
因为 5 万账号不是“放大 50 倍”这么简单，它会引入：
- 分区/分片
- 多区域调度
- 更细颗粒度的设备集群治理
- 更强的消息总线与数据湖分析

如果现在就为 5 万而设计完整架构，成本会很高，而且大概率会做错。

更好的路线是：

**先确保 1000 账号阶段边界清晰、执行面可横向扩容、存储可分区，再向 5 万演进。**

## 14. 需要提前做对的几个关键设计

## 14.1 项目级隔离必须从第一天做真隔离
不是 UI 隔离，而是：
- 表结构带 `project_id`
- 缓存命名空间隔离
- 向量库索引隔离
- 审计范围隔离
- 权限 scope 隔离

## 14.2 机器人与工作流版本解耦
不要让 Bot 直接等于一份可编辑流程。
应该是：
- Bot 引用一个当前生效 WorkflowVersion
- 草稿编辑不影响线上 Bot
- 发布后切换引用版本

## 14.3 渠道动作必须走适配器
工作流里表达的是“设置标签 / 拉群 / 发消息 / 取消托管”这些能力意图，
不是直接写死企业微信或微信个人号实现。

## 14.4 回复生成与发送执行要拆开
- 生成：LLM / Knowledge / Workflow Runtime
- 发送：Channel Adapter / Device App

这样才能做：
- 人工审核
- 风控拦截
- 重试补偿
- 多端回执

## 14.5 会话状态和客户状态要分离
- 会话状态：当前轮是否托管、是否转人工、当前工作流节点、等待状态
- 客户状态：长期标签、阶段、偏好、画像

很多系统就是把这两层揉在一起，最后会话逻辑非常乱。

## 15. 关键风险与架构提醒

## 15.1 最大风险不是 LLM，而是渠道执行稳定性
真正会拖垮系统的往往不是模型，而是：
- 设备掉线
- 账号异常
- 指令丢失
- 执行回执不一致
- 群/联系人同步延迟

所以“设备网关 + 回执机制 + 幂等命令”优先级很高。

## 15.2 防风险要做“降低风险”，不要做“绕过风险”
建议方向：
- 官方 API 优先时优先官方 API
- 手机端执行采用人工可确认模式、频控、时段控制
- 对群发、主动触达、连续跟进设置上限
- 所有高风险动作留痕并可关停

核心目标应该是 **合规、稳态、可审计**，而不是“无限自动化”。

## 15.3 多Agent 很容易过度设计
如果每个小能力都独立成 Agent，系统会迅速复杂化。
建议只有三类情况才做独立 Agent：
- 需要独立 Prompt/模型策略
- 需要独立观测指标
- 需要独立异步执行生命周期

否则优先做成普通节点或子流程。

## 15.4 工作流模板继承要处理版本锁定问题
模板被下游项目继承后，要支持：
- 固定版本
- 跟随更新
- 分叉编辑

否则模板体系后期会很痛苦。

## 16. 我给出的初步架构判断

### 最适合当前阶段的路线

**“多项目运营控制台 + 无代码工作流中台 + 多Agent执行运行时 + 手机边缘执行 App” 的四段式架构。**

### 最合适的实现策略

**控制面模块化单体，执行面事件驱动拆分，设备面独立演进。**

### 产品上的最核心对象

不是 Agent，而是这 6 个：
- Project
- ChannelAccount / Device
- Bot
- WorkflowVersion
- Conversation / Customer
- Campaign / SOP Task

### 技术上的真正核心壁垒

不是页面，而是这 4 个：
- 工作流运行时
- 渠道适配与设备指令系统
- 多Agent协作编排模型
- 运营数据闭环与调试回放能力

## 18. 一句话总结

**Morphix 的核心不是“做一个机器人”，而是“把多项目私域运营的销售执行网络，抽象成可配置、可编排、可托管、可扩展的产品系统”。**
