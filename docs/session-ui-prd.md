# 渠道会话管理页 UI 改造 — 增量 PRD

> 文档版本：v1.0（简单 PRD）
> 编写人：许清楚（产品经理）
> 日期：2025-07-23
> 关联页面：`src/pages/Channels/ChannelSessions.tsx`

---

## 0. 项目信息

- **项目**：Morphix 私域运营 AI 协同平台
- **技术栈**：前端 React 18 + Vite + TypeScript（端口 5183）；后端 FastAPI + SQLite（端口 2181）
- **改造范围**：渠道会话管理页（四栏：账号 / 会话列表 / 聊天 / 客户详情）的左栏与中栏交互改造
- **原始需求复述**（来自业务方，共 5 条）：
  1. 企业微信下拉框字体调小，与「全部 / 在线 / 离线」标签同字号。
  2. 左栏（账号列表）与中栏（会话列表）为一个整体区域，中间分隔线可左右拖动调宽。
  3. 同步改为自动：取消中栏顶部「同步」按钮；扫码添加账号成功后自动同步一次；每次进入页面自动增量同步。
  4. 搜索框「请输入会话名称」单独一行，置顶中栏。
  5. 中栏新增一行五个功能按钮：回到顶部 / 定位未读聊天 / 定位当前选中聊天 / 创建群聊 / 一键已读。

---

## 1. 产品目标（Product Goals）

- **G1 去手动、降操作成本**：将「手动点同步」转为「进入即同步 + 扫码即同步」，移除冗余按钮，让运营无需关心同步时机。
- **G2 提检索与定位效率**：置顶搜索框 + 回到顶部 / 未读定位 / 选中定位 / 一键已读等快捷操作，解决长列表下的导航与红点清理痛点。
- **G3 增组织灵活性**：左中栏宽度可拖拽自适应，并支持在平台内直接建群后即时会话，减少对外部设备端的依赖。

目标彼此正交：G1 管数据新鲜度，G2 管浏览效率，G3 管布局与新建能力。

---

## 2. 用户故事（User Stories）

- **US1**：作为私域运营，我希望进入渠道会话管理页时自动同步，这样不必每次手动点「同步」也能看到最新会话。
- **US2**：作为运营，我希望扫码添加账号成功后自动同步一次，这样新账号的资料与会话能立即就绪。
- **US3**：作为运营，我希望搜索框固定在最顶部且独占一行，这样能随时按名称快速找会话。
- **US4**：作为运营，我希望能一键回到顶部 / 跳到第一个未读 / 跳到当前选中项，这样长列表里不用手动滚动。
- **US5**：作为运营，我希望能一键把本地未读标为已读但不动原渠道未读，这样能清理红点又不打扰客户侧状态。
- **US6**：作为运营，我希望能直接勾选好友建群并马上聊天，这样不用切到微信 / iPad 端操作。
- **US7**：作为运营，我希望左栏和中栏的分隔线可拖拽，这样能按账号名 / 会话名长度自行调整两栏宽度。

## 4. UI 设计稿（文字描述）

整体维持四栏（左：账号 / 中：会话列表 / 右一：聊天 / 右二：客户详情），本次改造聚焦左栏与中栏。

**左栏（账号列表）**
- 顶部：渠道下拉（企业微信 / 微信 / …，R1 字体缩小）+ 账号搜索框 + 状态 Tab（全部 / 在线 / 离线）。
- 下方：账号项列表（不变）。
- 右边缘：一条可拖拽竖线（与中栏共用分隔条，R2）。

**左 ↔ 中 分隔**
- 两栏之间一条竖向分隔条，鼠标悬停变高亮；按住左右拖动改变左栏宽度，中栏自适应填充；设最小 / 最大宽度约束。

**中栏（自上而下）**
1. **搜索框行（独占一行，整行宽度）**：「请输入会话名称」输入框。
2. **筛选行**：阅读（全部 / 未读）、托管（全部 / 已托管 / 未托管）下拉；原「同步」按钮位置改为轻量同步状态小字（无按钮）。
3. **Tab 行**：会话 / 好友 / 群聊（不变）。
4. **功能按钮行（新增，5 个等宽按钮）**：
   - **回到顶部**：列表滚动定位到第一个会话项。
   - **定位未读**：定位并高亮 / 选中第一个 `unreadCount > 0` 的会话项。
   - **定位选中**：列表滚动到当前 `selectedSessionId` 所在项。
   - **创建群聊**：打开好友多选弹窗（见下）；确认后新建群并刷新群聊列表。
   - **一键已读**：批量清除本地未读状态（不调 iPad 协议、不动原渠道未读）。
5. **列表区**：会话 / 好友 / 群聊 列表（布局与现有 `.session-row` 一致，不变）。

**创建群聊弹窗**
- 遮罩 + 居中卡片，标题「创建群聊」。
- 好友列表（多选 checkbox，支持关键字搜索过滤），数据取自当前账号好友。
- 底部「确认 / 取消」；确认后 loading，成功后关闭弹窗并刷新「群聊」Tab，新群出现在列表且可点击打开会话直接聊天。

## 5. 数据 / 接口变更

### 前端
- `src/pages/Channels/ChannelSessions.tsx`：
  - 移除「同步」按钮与 `handleSync` 的按钮绑定（保留状态轮询与状态提示）。
  - 新增「页面挂载自动同步」effect（对 `selectedAccountId` 触发，非阻塞 + 状态轮询，建议节流）。
  - 搜索框提至独占行渲染。
  - 新增功能按钮行及逻辑：列表滚动定位（回到顶部 / 未读 / 选中）、打开建群弹窗、调用已读接口。
  - 左栏 + 中栏包裹为可拖拽容器（分隔条 + 宽度状态）。
- `src/api/client.ts`（channelsApi）：新增 `createGroup(accountId, payload)`、`markSessionsReadLocal(accountId, sessionIds?)`（命名待定）。

### 后端
- **新增 iPad 协议函数**（建议 `ipad_client.create_chatroom(uuid, wxids: list[str], room_name?)`）：调用设备 / iPad 建群协议。当前 `ipad_client.py` 无建群函数，需先确认协议侧可用性（见 Q4）。
- **新增路由 `POST /api/channels/{account_id}/groups`（或 `/create-room`）**：
  - 入参：好友 wxid 列表（及可选群名）。
  - 逻辑：调 `create_chatroom` → 落库（`channel_groups` / 对应 session）→ 返回新群会话。建议复用 `run_full_sync` 的群写入路径或新增 upsert。
- **新增路由 `POST /api/channels/{account_id}/sessions/read-local`（批量）**：
  - 仅将本地会话未读标记 / `unreadCount` 清零，**不调用** iPad `mark_as_read`，不改动原渠道未读。
  - 入参：sessionId 列表，或「全部（当前账号）」。
- **复用现有接口**：`POST /api/channels/{account_id}/sync`、`GET /api/channels/{account_id}/sync-status`。
- **建议评估**：新增「进入页自动增量同步」所需的 incremental（seq 增量）分支；现有 `ipad_client.sync_all_data(uuid, limit, seq)` 已支持 seq，但 `ipad_sync.trigger_sync` 当前走全量 `run_full_sync`（见 Q2）。
- **不改动**：`channel_hosting.poll_wecom` 现有「扫码成功自动同步」逻辑保留。

### 数据模型
- 现有 `channel_groups` 表已存在；新建群写入需确定 `group_type`（customer_group / internal_group）取值规则（见 Q4）。
- 本地未读：确认 `unreadCount` 来源（sessions 表字段 or 消息表聚合），以便「一键已读」准确清零并持久化（见 Q3）。

## 6. 待确认问题（Open Questions）

1. **R1 元素确认**：「企业微信下拉框」是否即左栏渠道下拉（`.channel-select-trigger`）？需 UI 截图最终确认目标元素与期望字号（以「全部 / 在线 / 离线」状态 Tab 为基准）。
2. **增量同步模式**：进入页「自动增量同步」应复用现有全量 `run_full_sync`，还是后端新增 seq 增量分支？若新增，需评估工作量与「增量」界定（仅新消息 or 含新联系人 / 群）。
3. **一键已读持久化**：本地未读清零是纯前端内存态（刷新后恢复）还是需后端落库持久化？影响是否必须新增 `read-local` 接口（倾向落库持久化）。
4. **建群协议可用性**：iPad 协议侧是否已有可用建群能力（如某设备协议支持 create_chatroom）？当前 `ipad_client.py` 无对应函数，需确认函数可用性、参数（成员 wxid 列表 / 群名）及新建群的 `group_type` 归属（客户群 or 内部群）。
5. **拖拽约束**：左栏最小 / 最大宽度取值（如 160–360px）；是否需跨会话持久化宽度（localStorage，R2-P2）。
6. **建群好友数据源**：弹窗好友复用中栏「好友」Tab 的 `listContacts` 接口，还是需新增分页 / 搜索接口？好友量大时是否需要搜索与分页。
7. **自动同步节流**：频繁进出页面是否会重复触发同步？是否需对「进入页面」做节流（如 30s 内不重复）以避免与后台互斥（`_sync_active` 标志）冲突。
