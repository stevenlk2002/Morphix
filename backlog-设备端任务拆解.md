# Morphix 设备端任务拆解清单

## 1. 文档目标

这份文档是在以下成果基础上继续下钻：

- `design-MVP研发任务拆解清单.md`
- `backlog-后端任务拆解.md`
- `backlog-前端任务拆解.md`
- `design-接口设计-总控编排与会话运行时.md`
- `openapi-总控编排与会话运行时.yaml`
- `prototype/index.html`

目标不是再讨论“手机端要不要做”，而是把**设备执行端 App** 真正拆成：

- 可以直接分配给设备端开发的任务包
- 可以和设备接入后端逐项联调的接口任务
- 可以按 Sprint 推进的实现顺序
- 可以验收，而不是“手机上大概能跑”的交付清单

一句话说：

**这份文档回答的是：Morphix 设备端团队明天开工，先做什么、云端依赖什么、什么程度算 MVP 可交付。**

---

## 2. 设备端范围说明

这里说的“设备端”，特指安装在渠道手机上的**设备执行端 App / Device Agent**，不是：

- 控制面 Web 控制台
- 运营人员使用的移动管理端
- 单纯的扫码登录壳页面

它在系统里的定位是：

> **服务端编排的边缘执行站点**

负责把服务端的结构化命令，转成设备侧的真实执行动作，并把执行事实、在线状态、消息事件、联系人/群变更等回传给平台。

也就是说，设备端不是“另一个后台”，而是 **Edge Runtime**。

---

## 3. 设备端职责边界

## 3.1 设备端必须承担的职责

### A. 身份与连接
- 设备注册 / 绑定
- 设备鉴权
- 在线心跳
- 设备基础信息上报

### B. 命令执行
- 拉取待执行命令
- 本地排队与串行/并行控制
- ACK / complete / fail 回执
- 执行超时与失败分类

### C. 事件上报
- 入站消息上报
- 本地执行结果上报
- 设备运行告警上报
- 基础日志与诊断信息回传

### D. 本地数据同步
- 联系人基础信息同步
- 群聊基础信息同步
- 账号在线状态同步
- 最近执行上下文缓存

### E. 安全与风控
- 只执行授权命令
- 限速与节流
- 高风险动作二次确认或阻断
- 本地敏感数据最小化存储

---

## 3.2 设备端不应该承担的职责

### 不应放在设备端的事情
- 工作流路由决策
- Bot 选择与 Policy Router 判断
- Agent 推理主逻辑
- 业务审计主存储
- 跨设备调度决策
- 复杂 CRM 画像计算

一句话：

**设备端负责执行，不负责决定业务策略。**

这条边界必须死守，否则手机端会长成一个难升级、难审计、难治理的“第二后端”。

---

## 4. 设备端 MVP 技术决策

基于当前目标，我建议设备端 MVP 先做以下决策：

| 决策项 | 选择 | 原因 |
|---|---|---|
| 设备平台 | Android 优先 | 设备编队管理、后台任务与诊断能力更现实，MVP 不建议双端齐开 |
| App 形态 | 原生 Android App / Device Agent | 边缘执行、心跳、日志、命令队列比跨端 UI 更重要 |
| 本地存储 | SQLite / Room | 适合命令队列、回执缓存、联系人轻量索引 |
| 云端交互 | REST 拉命令 + 回执上报 | 已有接口基线，复杂度最低 |
| 实时策略 | 轮询 + 心跳 | 比长连接更稳，更适合首版联调与排障 |
| 鉴权方式 | Device Token | 与现有设备接入层契约一致 |

### 额外说明

1. **MVP 不建议先做 iOS**
   - 不是技术信仰问题，而是边缘执行、后台运行、设备编队治理的现实约束问题。

2. **MVP 不建议先追求花哨 UI**
   - 设备端首先是执行站点，其次才是操作界面。

3. **优先使用合规、授权的接入方式**
   - 设备端设计目标应是“可管、可审计、可授权”，而不是试图绕过平台规则。

## 5. 设备端研发原则

## 5.1 先把命令闭环打通，再做复杂采集
优先级必须围绕：

`设备注册/绑定 -> 拉命令 -> ACK -> 完成/失败回执 -> 心跳 -> 控制面可见`

不要一开始就扑到：
- 全量通讯录采集
- 全量群关系建模
- 花哨设备大盘
- 自动批量运维

## 5.2 设备端必须本地可恢复
手机端天然不稳定：
- 网络会断
- App 会被系统挂起
- 命令会重复到达
- 回执会因为抖动丢失

所以本地必须具备：
- 命令幂等去重
- 本地待回执缓存
- 重试机制
- 崩溃恢复后续传

## 5.3 风控优先于“执行得快”
设备端不是命令越快越好，而是要：
- 可限速
- 可暂停
- 可冻结
- 可人工接管
- 可审计为什么没执行

## 5.4 先做“看得见问题”
MVP 阶段设备端最怕黑盒。

所以日志、状态、失败原因、最近命令、最近心跳时间，优先级比“界面好不好看”更高。

## 6. 端云职责边界

| 事项 | 服务端负责 | 设备端负责 |
|---|---|---|
| Bot / Workflow 选择 | 是 | 否 |
| 命令生成 | 是 | 否 |
| 命令鉴权 | 是 | 是（本地校验 token / deviceId） |
| 命令执行 | 否 | 是 |
| ACK / complete / fail 留痕 | 是 | 是（发起上报） |
| 消息事实存储 | 是 | 部分缓存 |
| 联系人/群同步主存 | 是 | 采集与增量上报 |
| 人工接管策略 | 是 | 配合执行 |
| 风控阈值配置 | 是 | 执行本地限流与阻断 |

### 关键原则

> **云端决定做什么，设备端决定如何安全地执行和如何如实回报。**

## 7. 原型到设备端能力映射

结合 `prototype/index.html`，可以提炼出设备侧与渠道账号相关的 MVP 能力：

| 原型意图 | 设备端能力映射 | 备注 |
|---|---|---|
| 渠道账号在线 / 离线 | 设备心跳与账号状态上报 | P0/P1 |
| 会话消息收发 | 入站消息上报 + 下行命令执行 | P0 |
| 创建群聊 / 群管理 | 结构化群命令执行能力 | P1 |
| 联系人 / 群列表查看 | 联系人与群基础同步 | P1 |
| 企业微信 / 微信渠道切换 | 设备绑定的 channel/account 元数据上报 | P0 |
| 工作流触达动作 | `send_message` / `send_media` 等命令执行 | P0 |
| 同步联系人 / 群聊 | `sync_contacts` / `sync_groups` 命令执行 | P1 |

需要注意：

**原型里看到的很多“渠道动作”在产品上看像页面能力，落地时其实一半都依赖设备端执行闭环。**

---

## 8. 设备端任务优先级定义

- **P0**：不做就无法完成设备侧 MVP 主链路
- **P1**：主链路能跑，但没有会明显影响联调、排障、可运营性
- **P2**：第二轮增强项

---

## 9. 推荐设备端模块划分

建议采用 feature-first 结构，而不是按“页面 / 工具类”散着堆：

- `app/bootstrap/`：启动、配置、依赖注入、环境切换
- `app/auth/`：设备绑定、token、设备身份管理
- `app/heartbeat/`：心跳与在线状态
- `app/commands/`：命令拉取、命令队列、命令执行器、回执上报
- `app/messages/`：入站消息采集与增量上报
- `app/contacts/`：联系人同步
- `app/groups/`：群聊同步
- `app/observability/`：日志、诊断、崩溃恢复、最近命令记录
- `app/risk-control/`：频控、暂停、冻结、人工接管协作
- `app/ui/`：最小必要管理页（绑定状态、在线状态、最近命令、诊断页）

# 10. P0 任务清单：设备主链路必须项

# 10.1 启动、身份与绑定

## DE-001 设备启动骨架与环境配置
- **优先级**：P0
- **Owner**：设备基础
- **目标**：建立 App 启动流程、环境配置、日志初始化、本地数据库初始化
- **依赖**：无
- **建议落点**：
  - `app/bootstrap/`
  - `app/observability/`
- **验收标准**：
  - Debug / Staging 环境可切换
  - 本地数据库可初始化
  - 启动日志可见

## DE-002 设备身份模型与本地安全存储
- **优先级**：P0
- **Owner**：设备基础
- **目标**：定义 `deviceId`、`channelAccountId`、`projectId`、`token` 等本地身份模型与安全存储
- **依赖**：DE-001、后端 BE-301
- **验收标准**：
  - 设备重启后身份可恢复
  - token 不明文散落在普通缓存

## DE-003 设备绑定页 / 初始化绑定流程
- **优先级**：P0
- **Owner**：设备基础 + 简易 UI
- **目标**：提供最小可用的设备绑定流程，让测试设备可以被纳入项目和渠道账号
- **依赖**：DE-002
- **说明**：当前 OpenAPI 初稿尚未正式收录“设备注册/绑定”接口，这部分应和设备接入后端一起补齐契约
- **验收标准**：
  - 设备可完成一次绑定
  - 绑定后本地能拿到有效身份信息
  - 设备端能展示当前绑定项目 / 账号 / 渠道

---

# 10.2 心跳与在线状态

## DE-101 心跳调度器
- **优先级**：P0
- **Owner**：设备基础
- **目标**：定时上报设备在线状态、App 版本、网络概况、最近活跃时间
- **依赖**：DE-002、后端 BE-506
- **接口对应**：下一轮补充 `/api/device/heartbeat`
- **验收标准**：
  - App 在线时按固定周期上报
  - 临时断网后恢复可继续上报

## DE-102 在线状态本地展示
- **优先级**：P0
- **Owner**：简易 UI
- **目标**：让测试和运维能在设备上直接看到“当前已在线 / 最近心跳 / 最近失败原因”
- **依赖**：DE-101
- **验收标准**：
  - 页面可见最近一次心跳时间
  - 页面可见最近一次心跳失败原因

---

# 10.3 命令拉取与本地队列

## DE-201 待执行命令拉取客户端
- **优先级**：P0
- **Owner**：命令模块
- **对应接口**：`GET /api/device/commands/pending`
- **依赖**：DE-002、后端 BE-302
- **目标**：实现设备按 `deviceId` 拉取待执行命令
- **验收标准**：
  - 能稳定拉取属于当前设备的命令
  - 异常返回可识别并记录

## DE-202 本地命令队列与去重机制
- **优先级**：P0
- **Owner**：命令模块
- **目标**：实现本地 `pending -> executing -> acked/done/failed` 队列与去重逻辑
- **依赖**：DE-201
- **验收标准**：
  - 重复拉到同一命令不会重复执行
  - App 重启后未完成命令仍可恢复

## DE-203 命令调度器与并发控制
- **优先级**：P0
- **Owner**：命令模块
- **目标**：定义同一渠道账号下命令的串行/并行策略，避免本地执行冲突
- **依赖**：DE-202
- **验收标准**：
  - 同一会话高冲突动作不会并发打架
  - 支持基本的执行超时控制

---

# 10.4 ACK / 完成 / 失败回执闭环

## DE-301 ACK 回执上报客户端
- **优先级**：P0
- **Owner**：命令模块
- **对应接口**：`POST /api/device/commands/{commandId}/ack`
- **依赖**：DE-201、DE-202、后端 BE-303
- **目标**：设备接单后尽快 ACK，缩短云端“悬空等待”时间
- **验收标准**：
  - 命令开始执行前完成 ACK 上报
  - ACK 失败会进入重试队列

## DE-302 完成回执上报客户端
- **优先级**：P0
- **Owner**：命令模块
- **对应接口**：`POST /api/device/commands/{commandId}/complete`
- **依赖**：DE-301、后端 BE-304
- **目标**：在本地动作完成后回传完成事实与结果摘要
- **验收标准**：
  - 完成回执能携带结构化结果
  - 回执成功后本地命令状态完成归档

## DE-303 失败回执上报客户端
- **优先级**：P0
- **Owner**：命令模块
- **对应接口**：`POST /api/device/commands/{commandId}/fail`
- **依赖**：DE-301、后端 BE-305
- **目标**：标准化上报失败事实，避免服务端只看到“超时”
- **建议失败分类**：
  - `network_unavailable`
  - `device_offline_transition`
  - `permission_missing`
  - `execution_timeout`
  - `manual_interrupted`
  - `channel_action_rejected`
- **验收标准**：
  - 失败原因可被结构化上报
  - 同一失败不会无限重试轰炸服务端

## DE-304 回执离线缓存与补传
- **优先级**：P0
- **Owner**：命令模块
- **目标**：解决“命令执行成功但回执没发出去”的经典断层
- **依赖**：DE-301、DE-302、DE-303
- **验收标准**：
  - 断网恢复后可自动补传回执
  - 不会把已成功回执反复补传成脏数据

# 10.5 最小执行能力

## DE-401 `send_message` 命令执行器
- **优先级**：P0
- **Owner**：执行模块
- **依赖**：DE-203、后端 BE-208
- **对应命令类型**：`send_message`
- **目标**：支持最小文本消息执行闭环
- **验收标准**：
  - 能处理一条标准文本消息命令
  - 能把结果走完 ACK -> complete / fail

## DE-402 `send_media` 命令执行器骨架
- **优先级**：P0
- **Owner**：执行模块
- **依赖**：DE-203
- **对应命令类型**：`send_media`
- **目标**：先把媒体类命令的结构与执行骨架建起来，即使首版只支持单一媒体类型
- **验收标准**：
  - 命令体结构明确
  - 不支持的媒体类型能明确 fail，而不是静默丢弃

## DE-403 本地执行审计记录
- **优先级**：P0
- **Owner**：执行模块 + 观测模块
- **目标**：记录最近 N 条命令的开始、结束、状态、错误码、耗时
- **依赖**：DE-301、DE-302、DE-303
- **验收标准**：
  - 设备本地可以查最近执行记录
  - 调试时能快速定位失败命令

---

# 10.6 与人工接管协作

## DE-501 人工接管暂停开关
- **优先级**：P0
- **Owner**：风险控制模块
- **目标**：当服务端会话进入人工接管态时，设备端停止自动执行该会话下的主动动作
- **依赖**：DE-201、后端 BE-405 / BE-406
- **验收标准**：
  - 被接管会话不会继续自动发送主动消息
  - 交还后可按服务端状态恢复

## DE-502 本地冻结与远程暂停能力
- **优先级**：P0
- **Owner**：风险控制模块
- **目标**：允许设备因本地异常、权限异常、人工暂停而进入冻结态
- **依赖**：DE-501
- **验收标准**：
  - 冻结态下不继续消化主动命令
  - 设备端有明确冻结原因可见

# 11. P1 任务清单：联调、可观测与运营可见性关键项

## DE-601 入站消息增量上报
- **优先级**：P1
- **Owner**：消息模块
- **目标**：把设备侧捕获到的新消息标准化上报到运行时入口
- **依赖**：DE-002、后端入站消息链路
- **价值**：形成真正的“手机 -> 服务端编排”正向入口
- **验收标准**：
  - 消息可带会话标识、时间戳、发送方基础信息
  - 重复消息可被去重

## DE-602 联系人基础同步
- **优先级**：P1
- **Owner**：联系人模块
- **对应命令类型**：`sync_contacts`
- **依赖**：DE-201、后端第二批增强表
- **目标**：同步联系人基础信息，而不是一开始就追求复杂画像
- **验收标准**：
  - 支持全量首刷 + 增量更新思路
  - 失败可重试，成功可记录游标

## DE-603 群聊基础同步
- **优先级**：P1
- **Owner**：群模块
- **对应命令类型**：`sync_groups`
- **依赖**：DE-201
- **目标**：同步群聊基础元数据，用于控制面列表与运营筛选
- **验收标准**：
  - 至少可同步群基础信息与成员数量等摘要字段

## DE-604 最近命令 / 最近错误诊断页
- **优先级**：P1
- **Owner**：简易 UI + 观测模块
- **目标**：给测试、实施、运维一个设备侧最小诊断页
- **依赖**：DE-403
- **验收标准**：
  - 可查看最近命令
  - 可查看最近错误码与最近心跳时间

## DE-605 本地日志导出能力
- **优先级**：P1
- **Owner**：观测模块
- **目标**：支持在调试阶段导出本地日志包
- **依赖**：DE-001
- **验收标准**：
  - 日志可按时间窗口导出
  - 不包含不必要的敏感明文

## DE-606 版本与升级信息展示
- **优先级**：P1
- **Owner**：设备基础
- **目标**：展示当前版本、构建号、配置版本、最近更新时间
- **依赖**：DE-001
- **验收标准**：
  - 实施和测试能确认当前设备版本

# 12. P2 任务清单：第二轮增强项

# 14. 设备端与后端联调顺序

## 联调 1：设备身份联调
- 目标：设备能绑定、能带身份访问、非法 token 被拒绝
- 对齐对象：DE-002 / DE-003 <-> BE-301
- 验收：
  - 正确 token 可通过
  - 错误 token 明确返回

## 联调 2：命令拉取联调
- 目标：设备可以稳定拉到自己的待执行命令
- 对齐对象：DE-201 / DE-202 <-> BE-302
- 验收：
  - 返回字段与 OpenAPI 一致
  - 重复拉取不重复执行

## 联调 3：ACK / 完成 / 失败链路联调
- 目标：把设备命令回执全链路打通
- 对齐对象：DE-301 / DE-302 / DE-303 / DE-304 <-> BE-303 / BE-304 / BE-305
- 验收：
  - `pending -> acked -> done/failed` 状态稳定
  - 重复回执不污染主状态机

## 联调 4：人工接管协同联调
- 目标：服务端一旦接管，设备端主动动作能停住；交还后可恢复
- 对齐对象：DE-501 / DE-502 <-> BE-405 / BE-406
- 验收：
  - 设备不再盲目继续自动动作
  - 恢复后不重复补发旧命令

## 联调 5：消息与同步联调
- 目标：设备侧入站消息、联系人、群聊能形成增量上报
- 对齐对象：DE-601 / DE-602 / DE-603 <-> 运行时入口与增强表
- 验收：
  - 至少能形成一条标准化入站消息和一条同步命令闭环

# 15. 设备端验收门禁

设备端 MVP 不是“手机上看起来有页面”就算完成，至少要过下面这些门：

## 15.1 基础可用门
- 设备能绑定
- 设备能心跳
- 设备能稳定拉命令
- 设备重启后身份与本地队列可恢复

## 15.2 命令闭环门
- 能处理至少一种文本消息命令
- 能完成 ACK / complete / fail 三类回执
- 重复回执不会破坏状态
- 断网恢复后可补传关键回执

## 15.3 风控协作门
- 人工接管后设备端自动动作可停
- 设备冻结时不会继续无脑执行
- 失败原因可结构化上报

## 15.4 可观测门
- 本地能看到最近命令
- 本地能看到最近错误
- 本地能看到最近心跳
- 服务端能看到设备在线状态

# 16. 当前契约缺口与补充建议

这次拆设备端 backlog 时，有一个很实际的结论：

**当前 OpenAPI 初稿已经覆盖了设备命令拉取与回执主链路，但还没有把“设备注册/绑定”“设备心跳”“联系人/群同步上报”完整纳入正式契约。**

建议下一轮优先补这几类接口：

1. `POST /api/device/register` 或等价绑定接口
2. `POST /api/device/heartbeat`
3. `POST /api/device/messages/inbound` 或统一入站消息上报接口
4. `POST /api/device/contacts/sync`
5. `POST /api/device/groups/sync`

这不是吹毛求疵，而是因为：

> **设备端如果没有正式契约，就很容易在联调阶段靠临时字段和口头约定硬拼，后面一定返工。**

# 17. 风险提示

## 风险 1：设备端做成“第二后端”
后果：逻辑分叉、升级困难、审计困难。

应对：
- 业务决策留在云端
- 设备端只保留执行与本地恢复逻辑

## 风险 2：命令与回执没有本地持久化
后果：一断网就丢状态，服务端看到的世界是假的。

应对：
- 必做本地命令队列
- 必做回执补传

## 风险 3：把高风险动作做成默认自动执行
后果：一旦异常，无法及时收口。

应对：
- 先做限速、冻结、人工接管协作
- 高风险动作逐项灰度放开

## 风险 4：设备端没有诊断页
后果：联调靠猜，排障靠缘分。

应对：
- 最小诊断页必须进 P1
- 最近命令 / 最近错误 / 最近心跳必须本地可见

# 18. 一句话结论

Morphix 的设备端 MVP，不是先做一个“很像 App 的 App”，而是先做一个：

> **可绑定、可在线、可接命令、可回执、可暂停、可排障的边缘执行站点。**

先把这条链路做稳：

`设备绑定 -> 在线心跳 -> 拉命令 -> ACK -> complete/fail -> 人工接管协作 -> 本地诊断`

等这条线稳了，再扩联系人同步、群动作、批量治理和 OTA，系统就不会从第一天开始长歪。
