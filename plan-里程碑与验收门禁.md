# Morphix 里程碑与验收门禁

## 1. 文档目标

这份文档不是重复路线图，而是把《`plan-MVP实施路线图.md`》里的阶段推进逻辑，进一步收敛成一套**可以拿来卡项目节奏、卡联调节奏、卡灰度上线节奏**的门禁体系。

它要解决的不是“大家接下来做什么”，而是：

- 什么条件满足了，才能进入下一阶段
- 什么问题出现了，必须先停下来
- 哪些结果算“完成”，哪些只是“看起来差不多”
- 谁来拍板，谁来提供证据，谁来承担阻塞项关闭责任

一句话说：

**这份文档回答的是：Morphix 的 MVP 到底什么时候能算阶段完成，什么时候绝对不能硬往下推进。**

---

本门禁文件覆盖以下对象：

- 控制面后端
- 运行时后端

- 控制台前端

## 3. 门禁设计原则

## 3.1 围绕业务闭环，而不是围绕代码数量

Morphix 的里程碑必须围绕这条主链路来验收：

`入站消息 -> 编排执行 -> 命令下发 -> 设备回执 -> 人工接管 -> 运行态与审计`

所以：

- “接口写完了”不等于阶段完成
- “页面做出来了”不等于阶段完成
- “模拟数据能演示”也不等于阶段完成

只有链路在目标阶段内**真实闭环**，才能算通过。

## 3.2 进入条件和退出条件必须分开

每个阶段都必须明确两类门：

- **进入门**：没满足就不能开工下一阶段
- **退出门**：没满足就不能宣布阶段完成

这样可以避免：

- 阶段开始时边界不清
- 阶段结束时靠口头“差不多”过门

## 3.3 验收必须有证据，不靠口述

每个门禁都必须对应一种可检查证据，例如：

- API 实测记录
- OpenAPI 校验结果
- migration 执行记录
- 控制台截图 / 操作录屏
- 设备日志 / 回执样本
- 压测报告 / 错误演练记录
- 灰度样本清单与回滚开关验证

## 3.4 阻塞项优先级高于排期压力

如果关键阻塞项未关闭：

- 不得因为“时间紧”强行推进
- 不得用临时人工操作掩盖系统缺口
- 不得把会影响主链路稳定性的缺陷留到灰度后再说

一句话：

**排期可以解释延期，门禁不能解释失守。**

## 4. 角色与职责

## 4.1 角色划分

| 角色 | 主要职责 |
|---|---|
| 产品 / 实施负责人 | 业务场景验收、灰度样本确认、人工接管流程确认 |
| 后端主负责人 | 运行时主链路、状态机、命令闭环、审计链路验收 |
| 前端负责人 | 控制台可观察性、可接管性、可排障性验收 |
| 设备端负责人 | 设备绑定、心跳、命令执行、回执、诊断能力验收 |
| 测试 / 联调负责人 | 组织联调、管理缺陷、出具阶段验证结论 |
| 项目负责人 | 决定阶段是否过门，协调阻塞项升级处理 |

## 4.2 决策原则

- **技术正确性**：由对应技术 owner 负责确认
- **业务可用性**：由产品 / 实施负责人确认
- **是否过门**：由项目负责人基于证据拍板
- **出现跨团队争议**：以主链路稳定性优先，而不是以局部功能完成度优先

---

## 5. 门禁状态定义

建议所有里程碑统一使用 4 种状态，而不是只说“完成/未完成”。

| 状态 | 含义 | 可否进入下一阶段 |
|---|---|---|
| Green | 所有 P0 门禁通过，P1 风险可控 | 可以 |
| Yellow | P0 基本通过，但有少量已知风险需带着清单推进 | 需项目负责人批准后才可以 |
| Red | 存在未关闭的 P0 缺陷或主链路断裂 | 不可以 |
| Blocked | 存在外部依赖未满足，当前团队无法单独关闭 | 不可以 |

### 关键规则

- **Red** 不能口头降级成 Yellow
- **Blocked** 必须显式标出 owner 和预计解除时间
- **Yellow** 必须附带风险清单和补救动作

---

## 6. 阻塞项定义

以下情况默认视为 **P0 阻塞项**：

1. 同一入站消息无法稳定去重
2. `workflow_run` 无法创建或状态不可信
3. `device_command` 能创建但设备无法稳定拉取
4. ACK / complete / fail 回执会破坏状态机
5. 人工接管后系统仍继续自动主动发送
6. 控制台无法定位运行态或审计事实缺失
7. 灰度时没有停用 Bot / 冻结设备 / 转人工开关
8. 关键错误无 requestId、无日志、无可追踪事实

以下情况默认视为 **P1 风险项**：

1. 页面交互体验粗糙，但不影响主链路操作
2. 联系人 / 群同步能力不完整，但不阻塞主链路闭环
3. 调试页信息不够美观，但核心运行轨迹可见
4. 个别非主路径接口尚未完全工程化

---

## 7. 里程碑总览

建议沿用路线图中的 5 个关键里程碑，并为每个里程碑定义独立门禁。

| 里程碑 | 名称 | 核心问题 |
|---|---|---|
| M1 | 开发基线冻结 | 团队是否可以基于同一套底座开工 |
| M2 | 运行时命令可生成 | 入站消息是否能稳定走到命令创建 |
| M3 | 设备回执闭环成立 | 命令是否真的被设备接收并驱动状态推进 |
| M4 | 控制台可操作 | 运营/实施是否能看、能接管、能排障 |
| M5 | 小流量灰度就绪 | 系统是否已经具备可控试运行条件 |

# 8. M1 门禁：开发基线冻结

## 8.1 目标

让所有研发角色站到同一条起跑线上，避免边做边改基础规则。

## 8.2 进入条件

- 需求理解、架构、接口与 backlog 文档已形成当前基线
- 各主线 owner 已明确
- 仓库目录边界已确定

## 8.3 退出门禁

### P0 必须通过

1. 后端、前端、设备端工程骨架已入库
2. `.env.example`、本地运行方式、依赖说明完整
3. PostgreSQL / Redis 开发环境可拉起
4. `openapi-总控编排与会话运行时.yaml` 可预览、可校验
5. `openapi-设备接入补充接口.yaml` 可预览、可校验
6. requestId、统一错误结构、幂等等基础约定冻结

### P1 建议通过

1. OpenAPI 预览方式纳入仓库脚本
2. 演示环境变量模板齐备
3. owner、Sprint 节奏、联调窗口已排出

## 8.4 验收证据

- 仓库目录截图或 README
- `.env.example` 清单
- 基础服务拉起记录
- OpenAPI 语法校验结果
- 新成员按文档拉起环境成功记录

## 8.5 不过门典型情形

- 每条线各自猜接口
- 环境启动方式仍靠口头传
- OpenAPI 还在频繁改字段且无冻结规则

---

# 9. M2 门禁：运行时命令可生成

## 9.1 目标

证明 Morphix 的运行时主脑已经成立，能够从入站消息推进到命令创建。

## 9.2 进入条件

- M1 已通过
- 第一批核心表 migration 已可执行
- 种子数据可支撑演示链路

## 9.3 退出门禁

### P0 必须通过

1. `POST /api/runtime/inbound-events/messages` 可接收并校验消息
2. 同一 `sourceMessageId` 重复提交不会重复入链
3. 会话定位 / 创建逻辑成立
4. `session_runtime` 可创建并推进基础状态
5. Policy Router 规则版可输出结构化决策
6. `workflow_run` 能创建
7. `device_command` 能创建并落库
8. 关键事实对象可追溯：`message / session_runtime / workflow_run / policy_decision / device_command`

### P1 建议通过

1. Agent Executor 基础结果可记录
2. 关键错误码已可区分业务失败与系统失败
3. `/ready` 能感知 DB / Redis 依赖健康

## 9.4 验收证据

- Postman / curl / 自动化脚本实测记录
- 数据库事实样本
- 一条完整演示链路的 requestId 串联记录
- 运行时日志样本

## 9.5 不过门典型情形

- 命令只能靠脚本手塞，不能由编排链路创建
- 会话状态不稳定，重复消息会打出脏 run
- 决策事实没有落库，后续无法审计

---

# 10. M3 门禁：设备回执闭环成立

## 10.1 目标

证明服务端创建的命令，已经可以被设备接收、执行、回执，并可靠驱动状态推进。

## 10.2 进入条件

- M2 已通过
- 设备拉命令与设备鉴权接口已具备实现基础
- 设备端最小命令队列骨架可运行

## 10.3 退出门禁

### P0 必须通过

1. 设备完成注册 / 绑定并拿到有效身份
2. 设备可稳定心跳上报
3. `GET /api/device/commands/pending` 可返回属于当前设备的命令
4. `POST /api/device/commands/{commandId}/ack` 可稳定成功
5. `POST /api/device/commands/{commandId}/complete` 可驱动 run / session 状态推进
6. `POST /api/device/commands/{commandId}/fail` 可记录结构化失败原因
7. 重复 ACK / complete / fail 不破坏状态机
8. 断网恢复后关键回执可补传
9. 人工接管后设备不会继续自动执行主动动作

### P1 建议通过

1. 设备基础诊断页可看最近心跳与最近错误
2. 设备冻结 / 暂停指令可被本地识别
3. 设备诊断快照上报链路可验证

## 10.4 验收证据

- 真机或模拟器联调记录
- ACK / complete / fail 样本日志
- 设备回执驱动状态变化前后对比
- 人工接管场景录屏或操作记录

## 10.5 不过门典型情形

- 设备能拉命令但回执不稳定
- complete 成功后 session_runtime 不更新
- 人工接管后系统还会继续出主动命令
- 设备断网恢复后产生脏回执或重复执行

---

# 11. M4 门禁：控制台可操作

## 11.1 目标

证明 Morphix 已经不仅“后端能跑”，而是运营、实施、测试、研发都能通过控制台完成观察、接管与排障。

## 11.2 进入条件

- M3 已通过
- 查询接口、审计接口、接管接口已可联调
- 前端应用壳、API client、错误处理骨架已落地

## 11.3 退出门禁

### P0 必须通过

1. 会话列表可按项目、账号、状态查询
2. 会话详情可查看消息流水
3. 当前运行态可见
4. 人工接管可发起
5. 人工交还可执行
6. `workflow_run / node_execution / policy_decision / agent_invocation` 至少可查到基础排障信息
7. 401 / 403 / 404 / 409 / 422 等关键错误有统一可理解反馈
8. 控制台查询结果与数据库事实一致

### P1 建议通过

1. 列表空态、错误态、加载态一致
2. 运行态轮询策略经过节流处理
3. 最近错误和最近命令在界面上便于定位

## 11.4 验收证据

- 控制台页面录屏
- 前后端联调清单
- 错误态截图
- 典型排障场景回放记录

## 11.5 不过门典型情形

- 页面能打开，但关键接口字段对不上
- 接管按钮可点，但后端状态没有真实变化
- 调试页存在，但看不到关键运行事实

---

# 12. M5 门禁：小流量灰度就绪

## 12.1 目标

证明系统已经具备“有限真实业务试运行”的条件，而不是只在开发环境能演示。

## 12.2 进入条件

- M4 已通过
- 真实业务样本与试运行范围已确认
- 灰度值守人与回滚责任人已明确

## 12.3 退出门禁

### P0 必须通过

1. 真实样本链路可跑通：入站消息 -> 编排 -> 设备执行 -> 回执 -> 审计
2. 至少一轮错误演练完成：设备离线、回执失败、人工接管、命令失败
3. 关键指标可看：
   - 入站消息处理成功率
   - 命令 ACK 成功率
   - complete/fail 比率
   - 设备在线率
   - 人工接管触发率
4. 至少具备 4 个控制开关：
   - 停用 Bot
   - 冻结设备
   - 限制会话范围
   - 强制转人工
5. 有明确回滚方案与停止灰度方案
6. 产品 / 实施确认灰度样本范围与人工兜底流程

### P1 建议通过

1. 联系人 / 群同步有基础可见性
2. 设备诊断日志上传可在值守期辅助定位问题
3. 常见失败原因 Top N 已可统计

## 12.4 验收证据

- 灰度方案清单
- 试运行样本名单
- 演练记录
- 指标看板或统计快照
- 回滚开关验证记录

## 12.5 不过门典型情形

- 没有人工值守流程就直接放真实流量
- 没有停用 Bot / 冻结设备开关
- 异常发生后只能靠研发登录服务器临时救火

## 13. 联调门禁

除了里程碑门，还建议单独维护联调门，防止跨团队“接口都写了但联不起来”。

## 13.1 联调 1：后端内部主链路门

通过条件：

- 入站消息到 `device_command` 链路成立
- requestId 可串起来
- 状态机无明显断裂

## 13.2 联调 2：设备闭环门

通过条件：

- 命令可拉取
- ACK / complete / fail 全部实测通过
- 重复回执 no-op 逻辑成立

## 13.3 联调 3：控制台查询门

通过条件：

- 会话列表 / 详情 / 运行态与实际数据一致
- 人工接管 / 交还能驱动真实状态变更

## 13.4 联调 4：调试审计门

通过条件：

- run / 节点 / 决策 / Agent 调用均能查到
- 至少能还原一条失败样本链路

## 13.5 联调 5：真实业务样本门

通过条件：

- 真实样本场景下主链路稳定
- 异常切人工路径经过演练

---

## 14. 灰度门禁补充规则

## 14.1 灰度前必须准备的开关

以下开关缺一不可：

1. 项目级停用开关
2. Bot 级停用开关
3. 设备冻结 / 暂停开关
4. 会话范围限制开关
5. 人工接管强制开关

## 14.2 灰度前必须准备的值守能力

1. 值守联系人名单
2. 值守时段
3. 升级路径
4. 失败后处理 SOP
5. 灰度停止条件

## 14.3 触发立即停灰的条件

出现以下任一情形，建议直接停止灰度：

1. 设备命令失败率异常升高且原因不明
2. 人工接管失效
3. 同一消息重复触发多次主动动作
4. 关键事实对象落库缺失
5. 灰度期间无法定位 requestId 对应链路

---

## 15. 推荐验收清单模板

每个里程碑评审时，建议统一输出一份 1 页验收单，字段如下：

| 字段 | 说明 |
|---|---|
| 里程碑 | 例如 M3 |
| 当前状态 | Green / Yellow / Red / Blocked |
| 通过时间 | 评审日期 |
| 评审人 | 项目负责人 + 相关 owner |
| P0 结论 | 逐项通过 / 不通过 |
| P1 风险 | 可带入下一阶段的风险清单 |
| 阻塞项 | 未关闭问题及 owner |
| 证据链接 | API 记录、截图、日志、录屏、报告 |
| 是否允许进入下一阶段 | 是 / 否 |
| 备注 | 限制条件与补救动作 |

## 16. 当前阶段的直接建议

基于现在已经产出的文档，我建议当前项目管理动作按这个顺序继续：

### 动作 A：把本文件作为统一评审模板
用于项目例会、联调例会和灰度评审，不再让每条线各说各话。

### 动作 B：给每个里程碑补一张证据清单
尤其是：
- M2 的运行时链路证据
- M3 的设备闭环证据
- M4 的控制台录屏证据
- M5 的灰度与回滚证据

### 动作 C：把阻塞项定义同步到 backlog 管理方式里
不要只记“待办任务”，还要记“当前不能过门的阻塞项”。

---

## 17. 一句话结论

Morphix 的 MVP 难点，从来不是“功能列表够不够长”，而是：

> **每个阶段能不能用明确证据证明：主链路更完整了、风险更可控了、系统更接近可灰度了。**

所以这份门禁文档的核心作用，就是把“差不多完成”这种危险表述，彻底替换成：

`进入条件明确 -> 退出条件明确 -> 验收证据明确 -> 阻塞项明确 -> 责任人明确`

这样项目才不会一边推进，一边失真。
