# Morphix MVP 研发任务拆解清单

## 1. 文档目标

这份清单的目标不是继续谈抽象设计，而是把已经收敛的内容真正拆成**可以开工、可以排期、可以联调、可以验收**的研发任务。

当前拆解基于以下成果：

- `report-原型驱动需求理解与初步架构建议.md`
- `design-总控编排层详细设计.md`
- `design-总控编排时序与状态图.md`
- `design-核心数据模型设计.md`
- `design-数据库表结构设计.md`
- `design-接口设计-总控编排与会话运行时.md`
- `openapi-总控编排与会话运行时.yaml`

一句话说：

**这份文档回答的是：Morphix 的 MVP 到底先做什么、谁先做、依赖什么、做到什么程度算交付。**

---

## 2. MVP 总目标

MVP 目标不是把整个 Morphix 一次性做满，而是先打通一条可运行的业务闭环：

> **入站消息 -> 总控编排 -> 工作流执行 -> Agent 调用 -> 设备命令下发 -> 设备回执闭环 -> 人工接管 -> 运行态查询与审计回放**

这条链路一旦打通，平台就具备最核心的“AI 托管私域会话”的产品雏形。

---

## 3. MVP 范围边界

## 3.1 MVP 必须覆盖的能力

### A. 控制面
- 项目级基础管理
- 机器人基础配置
- 工作流版本选择与运行绑定
- 会话列表与消息查看
- 当前运行态查看
- 人工接管与交还
- 基础审计查询

### B. 执行面
- 标准化入站消息接入
- Session Orchestrator 主调度
- Policy Router 规则版路由
- Workflow Runtime 执行
- Agent Executor 结构化调用
- Device Command Gateway 下发设备动作
- 设备 ACK / 完成 / 失败回执闭环

### C. 设备边缘面
- 单设备身份注册与鉴权
- 拉取待执行命令
- 上报 ACK / complete / fail
- 基础在线状态上报

### D. 运维与治理
- 基础日志
- requestId 链路追踪
- 错误码统一
- 健康检查
- 关键运行审计

---

## 3.2 MVP 明确暂不做满的能力

### 产品侧暂缓
- 完整多渠道同时上线（建议先聚焦 1 个主渠道）
- 复杂 CRM 画像体系
- 自动化营销活动全量编排
- 精细化报表中心
- 多层审批流
- 全量 RBAC 细粒度权限矩阵

### AI / 路由侧暂缓
- 全量 Supervisor Agent 常驻参与
- 自学习策略路由
- 动态预算优化
- 自动模型评分与在线实验平台
- 多 Agent 自主协商式编排

### 设备侧暂缓
- 海量批量 OTA 管理
- 多设备跨区域调度优化
- 复杂反作弊控制台
- 完整设备风控画像

一句话：

**MVP 先做“能稳稳跑起来”，不是先做“看起来什么都支持”。**

---

## 4. 推荐研发分阶段

建议按 6 个阶段推进，而不是按技术栈散着做。

## Phase 0：研发底座准备
目标：把仓库、环境、数据库、契约和联调方式定住。

## Phase 1：核心主骨架落库
目标：把主实体、主状态、主链路数据先落下来。

## Phase 2：总控编排主链路打通
目标：实现入站消息 -> 编排 -> 执行 -> 命令下发 -> 回执闭环。

## Phase 3：控制面 MVP 打通
目标：让运营/实施人员可以看会话、看运行态、转人工、查日志。

## Phase 4：设备侧 MVP 打通
目标：设备可注册、拉命令、回执、维持在线心跳。

## Phase 5：联调、压测、灰度上线
目标：完成可发布版本，而不是只完成代码。

# 8. 总控编排与运行时流

这是 MVP 的核心主战场。

## 8.1 Session Orchestrator MVP

### 任务
- 接收入站消息事件
- 对会话进行定位或创建
- 创建 / 更新 `session_runtime`
- 控制并发与会话锁
- 决定是否新开 run 或并入现有 run

### 依赖
- 6.3
- 7.1

### 验收口径
- 同一 `sourceMessageId` 不重复入链
- 能根据 `conversationId` 或来源键定位会话
- 并发消息不会把同一会话状态写乱

---

## 8.2 Policy Router（规则版）

### 任务
- 基于项目、Bot、会话状态、客户阶段做规则路由
- 输出：
  - botSelection
  - workflowVersionSelection
  - allowedAgentSet
  - modelProfile
  - interruptDecision
  - handoffDecision
  - supervisorNeeded
- 生成 `policy_decision` 留痕

### 依赖
- 8.1
- 7.1

### 验收口径
- 规则有配置来源
- 决策结果可追溯
- 风控阻断可返回清晰 reasonCodes

---

## 8.3 Workflow Runtime MVP

### 任务
- 启动 `workflow_run`
- 推进节点执行
- 记录 `node_execution`
- 支持 waiting / interrupted / completed / failed
- 与设备命令节点、Agent 节点打通

### 依赖
- 8.2

### 验收口径
- 一次 run 的轨迹可回放
- 节点失败可记录错误码
- 支持基础中断 / 恢复 / 取消

---

## 8.4 Agent Executor MVP

### 任务
- 对接底层 LLM 调用
- 强制结构化输入输出
- 记录 `agent_invocation`
- 接入基础超时、重试、失败降级
- 支持 2~4 类核心 Agent 能力

### MVP 建议先做的 Agent
- 在线问答 Agent
- 销售推进 Agent
- 风险兜底 Agent
- 表达控制 Agent

### 依赖
- 8.3

### 验收口径
- 每次 Agent 调用均有调用记录
- 能返回结构化结果，而不是纯文本黑盒
- 超时与失败不会把主链路拖死

---

## 8.5 Device Command Gateway

### 任务
- 创建设备命令
- 维护 `device_command` 状态
- 给轮询型设备提供待执行命令接口
- 支持 ACK / complete / fail 回执推进会话与运行态

### 依赖
- 8.3
- 7.1

### 验收口径
- 命令从 `pending -> acked -> done` 可正确流转
- 失败回执可触发重试 / 转人工 / 冻结策略中的至少一种
- 命令具备幂等性

---

## 8.6 Human Handoff Coordinator

### 任务
- 发起人工接管
- 中断 active run
- 标记 `session_runtime` 到 `HUMAN_HANDOFF`
- 交还托管后支持 `idle / continue / replan`
- 留痕 handoff 记录与 policy 决策

### 依赖
- 8.1
- 8.3

### 验收口径
- 接管期间 AI 不再主动发言
- 交还后能按模式恢复
- 审计日志中可追溯每次接管动作

---

## 8.7 运行态查询与审计能力

### 任务
- 查询 `session_runtime`
- 查询 `workflow_run`
- 查询 `node_execution`
- 查询 `policy_decision`
- 查询 `agent_invocation`

### 依赖
- 8.1 ~ 8.6

### 验收口径

## 9.4 设备端 MVP 形态建议

### 建议
MVP 阶段设备端不必一开始做成完整复杂 App，可按风险和资源选择：

### 方案 A：轻量设备代理
- 优先验证协议和闭环
- 开发速度快
- 适合内部联调

### 方案 B：最小可用设备 App
- 更接近真实生产形态
- 适合早期验证端到端体验
- 成本更高

### 架构建议
- **联调期先 A，试点期再逐步过渡到 B。**

## 11.2 主链路联调用例

### 必测链路
1. 新消息进入 -> 自动创建 / 定位会话 -> 启动 run
2. run 调用 Agent -> 创建设备命令 -> 设备 ACK -> complete
3. 设备失败回执 -> 运行态变更 -> 审计可查
4. 人工接管 -> AI 停止 -> 交还托管 -> replan 恢复
5. 重复消息 -> 幂等去重
6. 离线设备 -> 命令失败 / 风险阻断

### 依赖
- 8.x
- 9.x
- 10.x

### 验收口径
- 以上链路均可稳定复现
- 每条链路都有成功 / 异常两类用例

# 12. 推荐任务依赖顺序

建议按下面这个顺序推进最稳：

1. **基础底座**
2. **主骨架表 + migration**
3. **OpenAPI 对齐**
4. **Session Orchestrator**
5. **Policy Router（规则版）**
6. **Workflow Runtime**
7. **Agent Executor**
8. **Device Command Gateway**
9. **设备端命令拉取与回执**
10. **人工接管**
11. **运行态 / 审计查询接口**
12. **控制面会话页 / 详情页 / 调试页**
13. **联调与压测**
14. **灰度发布**

最不建议的推进方式是：

- 先做一堆复杂 Agent
- 先做一堆大屏报表
- 先做全量多渠道兼容
- 但主闭环还没打通

那种做法很容易做出一个“看起来功能很多，实际上跑不起来”的系统。

# 14. MVP 验收清单

只要以下 10 条都满足，MVP 就算真的成了，而不是“文档成了”。

1. 能通过接口接收标准化入站消息
2. 能自动定位或创建会话
3. 能基于规则选择 Bot 与 workflow_version
4. 能启动并推进一次 workflow_run
5. 能成功调用至少一类 Agent 并生成结构化结果
6. 能创建设备命令并被设备成功 ACK + complete
7. 设备失败时能反映到运行态与审计里
8. 人工可以接管并交还托管
9. 控制面能看到会话、消息、运行态、节点轨迹、策略日志
10. 核心链路具备日志、错误码、幂等和基础压测结果

# 15. MVP 里程碑建议

## 里程碑 M0：底座完成
- 环境、数据库、OpenAPI、公共中间件准备完毕

## 里程碑 M1：主链路后端跑通
- 不带复杂前端，也能通过接口把核心链路跑通

## 里程碑 M2：设备闭环跑通
- 命令下发、ACK、完成、失败闭环完成

## 里程碑 M3：控制面可操作
- 运营人员能看、能查、能接管

## 里程碑 M4：灰度试点可用
- 有可观测性、有压测基线、有回滚方案

# 16. 我对研发排期的直接建议

如果只给一句很实在的话：

**先把“系统编排闭环”做出来，再把“产品完整度”补上。**

具体来说：

- 第一优先级不是多漂亮的 UI
- 也不是多复杂的 Agent 组织
- 更不是一开始支持所有渠道
- 而是：
  - 会话进入系统后别丢
  - run 能推进
  - 命令能下去
  - 回执能回来
  - 人工能接住
  - 故障能查出来

这条链路稳了，Morphix 才有资格继续长成更大的平台。
