一句话说：

---

## 2. MVP 总目标

MVP 不追求“一次性覆盖全部私域运营能力”，而是优先交付一条真正可运行的业务主链路：

> **入站消息 -> 总控编排 -> 工作流执行 -> Agent 调用 -> 设备命令下发 -> 设备 ACK / complete / fail 回执 -> 人工接管 -> 运行态查询与审计回放**

只要这条链路稳定跑通，Morphix 就具备了第一版可灰度验证的产品雏形。

## 3. 实施范围与非目标

## 3.1 本路线图覆盖的范围

### A. 后端
- 控制面后端
- 运行时后端
- 设备接入后端
- 公共底座（DB / Redis / 配置 / 日志 / OpenAPI）

### B. 前端
- 控制面 Web 控制台
- 会话工作台
- 运行态 / 调试页
- 人工接管与交还
- Bot / WorkflowVersion 绑定

### C. 设备端
- 设备执行端 App / Device Agent
- 设备绑定
- 命令拉取
- ACK / complete / fail 回执
- 心跳与本地诊断

### D. 联调与上线
- OpenAPI 对齐
- 主链路联调
- 验收门禁
- 小流量灰度上线

## 3.2 本路线图明确不追求一次做到的能力

- 全渠道一次性上线
- 全量 CRM 精细画像
- 全量自动化运营活动体系
- 全量 Supervisor Agent 常驻协作
- 海量设备编队治理与 OTA 平台
- 报表中心、复杂权限中心、全景运营看板

一句话：

**先把发动机装稳，不先把车漆喷满。**

---

## 4. 实施总原则

## 4.1 先主链路，后外围能力
严格围绕：

`入站消息 -> 编排执行 -> 设备命令 -> 回执闭环 -> 人工接管 -> 审计查询`

不要让以下能力抢走第一优先级：
- 大而全控制台
- 复杂营销能力
- 高级智能路由
- 花哨可视化

## 4.2 先确定性，后智能化
MVP 中：
- 总控以系统编排层为主
- Policy Router 先规则版
- Agent 先结构化调用
- Supervisor Agent 只灰度参与

## 4.3 先事实留痕，后体验打磨
以下事实必须先能沉淀：
- message
- session_runtime
- workflow_run
- node_execution
- policy_decision
- agent_invocation
- device_command
- handoff_record / 运行事件

## 4.4 先单渠道闭环，后多渠道扩张
建议 MVP 先聚焦 1 个主渠道，把：
- 会话闭环
- 命令闭环
- 人工接管闭环
- 基础同步闭环

做稳之后，再扩更多渠道。

## 5. 团队分工建议

建议按 4 条主线组织 owner，而不是所有人围着一个总 backlog 打转。

| 主线 | 主要职责 | 对应文档 |
|---|---|---|
| 公共底座 / 后端基础 | 配置、DB、Redis、鉴权、错误结构、OpenAPI、迁移 | `backlog-后端任务拆解.md` |
| 运行时与控制面后端 | 入站消息、编排、运行时、人工接管、查询与审计接口 | `backlog-后端任务拆解.md` |
| 控制台前端 | 会话工作台、运行态、调试页、Bot 绑定、鉴权与 API 集成 | `backlog-前端任务拆解.md` |
| 设备端与设备接入协同 | 设备绑定、命令拉取、回执、心跳、同步与本地诊断 | `backlog-设备端任务拆解.md` |

### 建议角色配置
- 1 名后端主负责（偏运行时）
- 1 名后端协同（偏控制面/设备接入）
- 1 名前端负责人
- 1 名设备端负责人
- 1 名测试 / 联调负责人（可由后端或前端兼任早期职责）
- 1 名产品/实施负责人做原型、场景验收与灰度反馈闭环

## 6. 跨团队总依赖图

建议把依赖理解成 3 条横向线 + 1 条纵向合流线：

### 横向线 A：后端基础线
`配置 -> DB/Redis -> migration -> OpenAPI 对齐 -> 统一错误/幂等`

### 横向线 B：运行时主链路线
`入站消息 -> 会话定位 -> Policy Router -> Workflow Runtime -> Agent Executor -> Device Command`

### 横向线 C：设备与控制台接入线
- 控制台：`会话列表 -> 会话详情 -> 运行态 -> 人工接管 -> 调试审计`
- 设备端：`设备绑定 -> 拉命令 -> ACK -> complete/fail -> 心跳 -> 本地诊断`

### 纵向合流线：主链路联调
`运行时命令创建 + 设备拉取/回执 + 控制台运行态可见 + 人工接管可切换`

只有纵向这条线合上，MVP 才算真正进入可验证状态。

## 7. 推荐实施阶段

建议整体分 6 个阶段推进。

---

# Phase 0：启动与基线冻结

## 目标
把研发底座和协作基线定住，避免后续边做边改基础规则。

## 关键产出
- 仓库目录边界冻结
- `.env.example` 与环境变量清单
- PostgreSQL / Redis 开发环境拉起
- OpenAPI 初稿成为唯一接口基线
- 任务 owner 与 Sprint 节奏确定

## 涉及主任务
- 后端：BE-001 ~ BE-005
- 前端：FE-001 ~ FE-003（可并行起壳）
- 设备端：DE-001（启动骨架）

## 进入条件
- 设计文档已形成基线
- 团队 owner 已明确

## 退出条件
- 新成员可以按文档拉起环境
- OpenAPI 可预览、可校对
- 各端工程结构已落仓库

## 风险
- 一开始没有统一契约
- 各端自己猜字段

## 应对
- 明确：OpenAPI 是主基线，设计文档是解释层

---

# Phase 1：数据骨架与系统底座落地

## 目标
把主实体、主状态、主事实对象先落库，先保证“系统有骨架”。

## 关键产出
- 第一批 migration
- 演示种子数据
- DB/Redis 健康检查
- requestId / 错误结构 / 幂等中间件

## 涉及主任务
- 后端：BE-101、BE-102、BE-003、BE-004
- 前端：FE-002、FE-004、FE-005（可并行做壳层能力）
- 设备端：DE-002、DE-003（身份模型与绑定流程）

## 进入条件
- Phase 0 完成

## 退出条件
- 核心表可建库
- 演示数据可支持一条测试链路
- 鉴权、错误结构、幂等规范已统一

## 风险
- 表先随手建，后面接口反推字段

## 应对
- 所有核心表以既有设计稿与 OpenAPI schema 对照后再落 migration

---

# Phase 2：运行时主链路打通

## 目标
让服务端先“动起来”，形成不依赖前端和真实设备也可验证的运行时主链路。

## 关键产出
- 入站消息接口
- 会话定位 / 创建逻辑
- Session Orchestrator
- Policy Router（规则版）
- Workflow Runtime
- Agent Executor
- Device Command 创建能力

## 涉及主任务
- 后端：BE-201 ~ BE-208
- 前端：FE-003、FE-004 持续对齐错误结构与查询模型
- 设备端：DE-201 / DE-202 可开始接 pending 命令 mock

## 进入条件
- 数据骨架已落地

## 退出条件
- 能通过接口跑通：
  `入站消息 -> 会话定位 -> 策略路由 -> workflow_run -> device_command`
- 命令事实能落库
- 关键状态可查询

## 风险
- 过早引入太多智能策略
- 运行时逻辑藏进控制器/脚本里

## 应对
- 严格按 Session Orchestrator / Policy Router / Runtime 边界落实现

---

# Phase 3：设备闭环与人工接管打通

## 目标
把服务端主链路从“创建命令”推进到“命令真正闭环”，并把人工接管加进来。

## 关键产出
- 设备鉴权
- 待执行命令拉取
- ACK / complete / fail 回执
- 回执驱动运行态推进
- 人工接管 / 交还
- 设备心跳初版

## 涉及主任务
- 后端：BE-301 ~ BE-305、BE-405、BE-406、BE-506
- 设备端：DE-101、DE-201 ~ DE-304、DE-401、DE-501、DE-502
- 前端：准备接入运行态与接管交互所需接口

## 进入条件
- 运行时可以稳定创建命令

## 退出条件
- 能跑通：
  `命令创建 -> 设备拉取 -> ACK -> complete/fail -> session_runtime 更新`
- 人工接管后 AI 主动动作可停住
- 设备在线状态基础可见

## 风险
- 回执状态机不严
- 重复 ACK / complete 污染主状态

## 应对
- 强制幂等
- 明确状态转移表
- 重复回执按 no-op 处理而不是报错打炸链路

---

# Phase 4：控制台 MVP 打通

## 目标
让运营、实施、测试和研发终于“看得见系统在干嘛”，形成最小可用控制台。

## 关键产出
- 会话列表页
- 会话详情工作台
- 消息流水面板
- 当前运行态面板
- 人工接管 / 交还操作
- Run / 节点 / 策略 / Agent 调用调试页
- Bot / WorkflowVersion 绑定页

## 涉及主任务
- 前端：FE-101 ~ FE-206、FE-301 ~ FE-305、FE-401 ~ FE-403（按 backlog 顺序推进）
- 后端：BE-401 ~ BE-406、BE-501 ~ BE-505
- 设备端：DE-102、DE-403、DE-604（本地诊断支撑联调）

## 进入条件
- 设备闭环与运行态推进已成立

## 退出条件
- 控制台能完整查看会话状态
- 运营可以发起人工接管与交还
- 研发/实施能定位 run、节点、决策、Agent 调用

## 风险
- 前端太早追求“还原全量原型”

## 应对
- 只围绕会话工作台、运行态与调试页做 MVP 页面

---

# Phase 5：联调、压测、灰度上线

## 目标
从“代码能跑”推进到“版本可发”。

## 关键产出
- 主链路联调完成
- 基础压测结果
- 关键错误演练
- 灰度策略
- 上线门禁清单

## 涉及主任务
- 后端：P1 联调/排障任务与心跳增强
- 前端：错误态、空态、轮询策略、调试体验完善
- 设备端：消息增量上报、联系人/群基础同步、日志导出

## 进入条件
- 控制台 MVP 可用
- 设备闭环稳定

## 退出条件
- 小流量可灰度
- 关键链路监控可看
- 异常时可回滚、可停机、可转人工

## 8. 里程碑建议

建议至少设 5 个里程碑，而不是只按日期推项目。

| 里程碑 | 名称 | 核心判断标准 |
|---|---|---|
| M1 | 开发基线冻结 | 环境、迁移、OpenAPI、目录边界确定 |
| M2 | 运行时命令可生成 | 入站消息到 `device_command` 主链路成立 |
| M3 | 设备回执闭环成立 | ACK / complete / fail 可驱动运行态推进 |
| M4 | 控制台可操作 | 会话、运行态、接管、调试可用 |
| M5 | 小流量灰度就绪 | 主链路联调通过、监控可见、门禁通过 |

### 关键提醒

不要把“页面做完了”当里程碑，也不要把“接口写完了”当里程碑。

**里程碑应该围绕可验证业务能力，而不是围绕代码数量。**

---

## 9. 推荐联调顺序

联调不要全量同时开火，建议按 5 步走。

## 联调 1：后端内部主链路联调
目标：不接前端、不接真实设备，也能验证入站消息到命令生成。

链路：
`runtime inbound -> conversation -> session_runtime -> policy -> workflow_run -> device_command`

## 联调 2：设备命令闭环联调
目标：先用假设备或模拟器跑通命令闭环。

链路：
`device_command -> pending pull -> ack -> complete/fail`

## 联调 3：控制台查询与接管联调
目标：前端能看会话、看运行态、发起接管。

链路：
`conversation query -> runtime query -> handoff -> return`

## 联调 4：调试与审计联调
目标：能看 run、节点、策略、Agent 调用。

链路：
`workflow_run -> node_execution -> policy_decision -> agent_invocation`

## 联调 5：灰度前完整业务联调
目标：用真实业务样本跑完整闭环。

链路：
`真实入站消息 -> AI 编排 -> 设备执行 -> 用户侧结果 -> 人工接管兜底 -> 审计回放`

## 10. 各阶段验收门禁

## 10.1 开发基线门
- 环境可拉起
- OpenAPI 可预览可校对
- migration 可执行
- requestId / 错误结构 / 幂等规范已统一

## 10.2 运行时门
- 同一 sourceMessageId 不重复入链
- 可生成 `workflow_run`
- 可生成 `device_command`
- 关键事实对象可留痕

## 10.3 设备闭环门
- 设备能拉命令
- ACK / complete / fail 状态可稳定流转
- 重复回执不破坏状态机
- 断网恢复后可补传关键回执

## 10.4 控制台门
- 会话列表可查
- 会话详情可看
- 当前运行态可见
- 人工接管可执行
- run / 节点 / 决策 / Agent 调用可排障

## 10.5 灰度上线门
- 小流量样本跑通主链路
- 关键错误有可观察告警
- 人工接管可兜底
- 设备异常可识别
- 至少准备一套回滚/停用策略

## 11. 灰度上线建议

MVP 上线不要“全量开闸”，建议按 3 级灰度推进。

## Level 1：研发自测灰度
- 固定测试项目
- 固定 Bot / WorkflowVersion
- 固定测试设备
- 使用演示数据与内部样本消息

## Level 2：实施小范围验证
- 固定 1 个业务项目
- 固定少量渠道账号
- 有明确人工值守
- 重点看：命令闭环成功率、接管时延、错误类型

## Level 3：业务试运行
- 扩到可控范围内的真实会话
- 控制 Bot 范围和会话范围
- 保留人工接管强兜底

### 灰度指标建议
- 入站消息处理成功率
- 命令 ACK 成功率
- complete/fail 回执比率
- 平均运行时延
- 人工接管触发率
- 设备在线率
- 设备命令失败原因 Top N

## 12. 关键风险与应对

## 风险 1：后端、前端、设备端各自推进，但没有统一节奏
后果：大家都在忙，但没有一条线真正可用。

应对：
- 以里程碑为中心推进
- 每个 Sprint 至少绑定一个“可联调目标”

## 风险 2：过早做复杂智能化
后果：系统不稳、成本高、问题难定位。

应对：
- MVP 阶段坚持规则版 Policy Router
- Supervisor Agent 只灰度参与

## 风险 3：设备侧契约不完整
后果：联调靠口头约定，后面返工。

应对：
- 尽快补齐：设备注册/绑定、心跳、联系人/群同步上报契约

## 风险 4：控制台做得很全，但主链路并不稳
后果：看起来热闹，实际上不可用。

应对：
- 所有页面优先服务主链路排障
- 不围绕“后台完整度”排优先级

## 风险 5：缺乏灰度与回滚方案
后果：一上线就只能硬扛。

应对：
- 上线前必须准备：停用 Bot、冻结设备、切人工、限制会话范围 的开关

## 14. 一句话结论

Morphix 的 MVP 推进，不应该按“前端先做完、后端再补、设备端最后接”这种传统瀑布思路来走。

更合适的方式是：

> **以主链路为纲，以里程碑为节拍，让后端、前端、设备端围绕同一条闭环逐阶段合流。**

先把这条线跑稳：

`入站消息 -> 编排执行 -> 命令下发 -> 设备回执 -> 人工接管 -> 运行态与审计`

等这条线稳了，Morphix 才是真正站起来了，后面的多渠道、复杂运营、设备规模化治理，才有往上叠的基础。
