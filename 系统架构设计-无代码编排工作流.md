# 系统架构设计 - 无代码编排工作流

> **版本**：v1.0 | **日期**：2026-07-11 | **状态**：初稿

---

## 一、文档目标

本文档描述无代码编排工作流子系统在多项目、多渠道私域运营平台中的总体架构、模块边界、核心链路、治理机制与扩展策略，为后续技术设计和实现提供统一蓝图。

---

## 二、架构定位

无代码编排工作流子系统位于平台能力中台层，向上承接运营、实施、产品、AI 配置人员的流程设计与发布需求，向下连接知识库、素材库、大模型、渠道适配器、调试监控和权限审计系统，形成从“流程定义”到“流程执行”的闭环。

其核心职责包括：

1. 提供可视化流程定义能力
2. 管理节点协议与节点配置
3. 管理工作流版本和发布生命周期
4. 提供运行时流程调度与节点执行能力
5. 提供变量上下文、知识检索、渠道动作适配能力
6. 提供调试、监控、审计、权限治理能力

## 四、核心模块设计

### 4.1 编排工作台（前端）

职责：
- 画布渲染与节点拖拽
- 连线编辑与类型校验
- 节点参数配置面板
- 视图管理（缩放、锁定、归位、缩略图）
- 流程调试入口、版本比较入口、发布入口

关键要求：
- 支持中大型流程图顺畅渲染
- 维护本地草稿态和服务端保存态
- 在 UI 层提前执行结构校验和类型校验

### 4.2 流程定义服务

职责：
- 保存工作流草稿
- 管理节点、边、布局信息
- 管理工作流元数据（名称、描述、所属项目、所属机器人）
- 提供节点参数 schema 给前端渲染

输出对象：
- WorkflowDefinition
- NodeInstance
- EdgeDefinition
- LayoutSnapshot

### 4.3 节点注册中心

职责：
- 管理系统所有节点定义
- 统一维护节点类型、分类、输入输出、配置 schema、运行执行器映射
- 支持内建节点、脚本型节点、渠道适配型节点三类

设计要点：
- 节点定义与节点实例分离
- 节点升级支持版本化
- 节点 schema 改变时需保证兼容性控制

### 4.4 版本发布服务

职责：
- 生成可发布版本快照
- 执行发布前校验
- 支持发布、下线、回滚
- 管理版本备注、差异说明、发布目标

设计要点：
- 发布版本必须是不可变快照
- 回滚本质上是将历史快照重新标记为当前生效版本
- 与机器人托管系统联动，同步运行版本

### 4.5 调试会话服务

职责：
- 提供“对话测试”入口
- 保存测试输入、上下文、节点执行轨迹
- 返回逐节点输入输出详情
- 支持上下文清空和重新测试

设计要点：
- 调试数据与生产运行日志分开存储
- 调试支持富媒体输入（文字、语音、附件）
- 支持以版本维度回放

### 4.6 工作流执行引擎

职责：
- 加载工作流发布版本
- 按依赖关系驱动节点执行
- 管理上下文状态、并发与排队
- 管理异常、重试、降级和终止策略

执行过程：
1. 接收触发事件
2. 初始化运行上下文
3. 找到所有可执行入口节点
4. 调用节点执行器
5. 收集输出并传播到下游
6. 当下游条件满足时加入执行队列
7. 输出最终结果并落日志

### 4.7 节点执行器框架

职责：
- 按节点类型分派执行器
- 为不同节点提供统一执行接口
- 对外隐藏脚本、HTTP、知识检索、渠道调用等差异

统一接口建议：

```yaml
NodeExecutor:
  execute(input, config, context) -> output
```

执行器分类：
- BuiltinExecutor：用户输入、文本包装、判断器等内建节点
- AIExecutor：AI 对话节点
- KnowledgeExecutor：知识检索类节点
- ScriptExecutor：脚本运行节点
- AdapterExecutor：渠道标签、拉群等动作节点

### 4.8 上下文管理器

职责：
- 管理运行中的全局变量、局部变量和客户属性快照
- 管理对话历史和临时中间结果
- 提供统一变量解析和模板替换能力

上下文层级：
1. 系统级默认变量
2. 工作流级全局变量
3. 节点级局部变量
4. 客户属性级持久状态
5. 调试/运行会话级临时变量

### 4.9 知识与检索层

职责：
- 管理普通知识库检索
- 管理高级知识库结构化筛选
- 提供统一的检索抽象

检索模式：
- 语义检索
- 全文检索
- 混合检索
- 结构化字段筛选 + 向量召回

设计要点：
- 普通知识库适合文档型导入和段落切片
- 高级知识库适合表格型、字段化数据
- 检索结果需标准化为知识引用对象，供 AI 节点消费

### 4.10 AI 模型接入层

职责：
- 屏蔽多模型差异
- 管理 Prompt 注入、输入拼装、结果解析
- 提供超时控制、重试、熔断和降级

关键能力：
- 模型路由
- Token 限制
- 输出文本清洗
- 敏感词过滤前后置处理

### 4.11 渠道适配层

职责：
- 将工作流中的动作节点转为渠道侧实际执行命令
- 适配不同渠道的标签、群组、托管和消息能力

适配对象：
- 企业微信
- 微信个人号
- WhatsApp 个人号
- WhatsApp 企业号
- 平台内部标签系统（Morphix 标签）

设计要点：
- 使用 Adapter Pattern
- 节点配置抽象能力层，适配器做渠道差异翻译
- 对授权不足、托管未开启等前置条件统一报错

### 4.12 权限治理服务

职责：
- 鉴权：用户是否可访问项目和流程
- 授权：用户是否可执行编辑、发布、回滚、删除等操作
- 审批：高风险发布和跨项目模板复制可选审批流

关键能力：
- RBAC + Project Scope
- 高危动作二次确认
- 关键资源最小权限原则

### 4.13 审计与监控服务

职责：
- 采集操作审计日志
- 采集运行日志和错误事件
- 形成节点级、工作流级、版本级监控指标

观测对象：
- 发布次数、回滚次数
- 调试成功率、运行成功率
- 节点失败率、平均耗时
- 知识检索命中率
- 渠道动作失败率

## 五、关键流程架构

### 5.1 编排保存流程

```text
前端编排工作台
  -> 本地结构校验
  -> 提交流程定义服务
  -> 服务端校验（必填/类型/引用）
  -> 保存草稿版本
  -> 返回保存结果
```

### 5.2 发布流程

```text
用户点击发布
  -> 权限治理服务校验发布权限
  -> 版本发布服务执行校验
  -> 生成 WorkflowVersion 快照
  -> 写入发布记录与审计日志
  -> 同步运行时生效版本
  -> 返回发布成功
```

### 5.3 调试流程

```text
用户进入对话测试
  -> 自动保存当前草稿
  -> 创建调试会话
  -> 工作流执行引擎按草稿/指定版本运行
  -> 记录节点输入输出
  -> 返回最终响应和调试详情
```

### 5.4 生产运行流程

```text
渠道事件触发
  -> 路由到对应项目/机器人
  -> 加载当前已发布版本
  -> 初始化上下文
  -> 执行引擎运行节点
  -> AI/知识/渠道适配器协同执行
  -> 输出结果回渠道
  -> 写入运行日志与指标
```

## 六、多项目与多渠道架构原则

### 6.1 多项目隔离

隔离对象：
- 工作流定义
- 发布版本
- 知识库
- 素材库
- 调试会话
- 运行日志
- 权限和审计范围

实现方式：
- 所有核心表带 `project_id`
- 缓存和消息主题按项目做命名空间隔离
- API 查询默认必须带项目上下文

### 6.2 多渠道扩展

原则：
- 工作流定义层不直接绑定单一渠道实现
- 节点表达的是“能力意图”，适配器负责翻译为渠道命令
- 不同渠道支持能力差异时，由节点配置层约束可用选项

## 七、版本与治理设计

### 7.1 生命周期

- 草稿 Draft
- 待发布 Ready
- 已发布 Published
- 已回滚 RolledBack
- 已归档 Archived

### 7.2 版本模型要求

每次发布应固化：
- 流程结构
- 节点配置
- 子流程引用版本
- 知识库映射关系
- 渠道适配配置
- 发布人、发布时间、备注

### 7.3 审计要求

记录：
- 谁在什么时候修改了什么
- 发布前后版本差异
- 回滚来源版本和目标版本
- 高权限动作审批链路

## 八、异常与容错策略

| 场景 | 策略 |
|------|------|
| AI 超时 | 重试 1 次，失败则进入兜底话术或人工兜底 |
| 知识检索无结果 | 返回空知识结果并由 AI 节点兜底 |
| JSON 解析失败 | 记录异常并中断依赖该输出的分支 |
| 渠道动作失败 | 标记失败并写运行日志，按配置决定重试 |
| 子流程不可用 | 阻断发布或在运行前校验拒绝执行 |

## 九、技术选型建议

### 9.1 前端
- React / Vue 任一成熟框架
- 流程图编辑引擎：React Flow / LogicFlow / X6
- JSON Schema 驱动配置面板

### 9.2 后端
- Java / Go / Node.js 任一服务化栈
- 工作流定义与发布服务独立部署
- 执行引擎可单独弹性扩容

### 9.3 存储
- 关系型数据库：MySQL / PostgreSQL
- 缓存：Redis
- 对象存储：COS / S3
- 向量数据库：Milvus / pgvector / Elasticsearch 向量检索

### 9.4 消息与监控
- MQ：Kafka / RabbitMQ
- 监控：Prometheus + Grafana
- 日志：ELK / Loki
