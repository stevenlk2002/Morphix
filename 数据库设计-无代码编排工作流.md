# 数据库设计 - 无代码编排工作流

> **版本**：v1.0 | **日期**：2026-07-11 | **状态**：初稿

---

## 一、设计目标

数据库设计需要支持以下目标：

1. 支持多项目数据隔离
2. 支持工作流定义、版本发布、回滚与审计
3. 支持节点实例化配置、连线关系、变量体系
4. 支持知识库、素材库、子流程与嵌入式机器人能力联动
5. 支持调试记录与生产运行日志
6. 支持权限与治理场景

---

## 二、设计原则

1. **项目隔离优先**：核心业务表统一带 `project_id`
2. **定义与实例分离**：节点定义不等于节点实例
3. **草稿与发布分离**：编辑态和运行态采用不同存储模型
4. **审计可追踪**：关键资源修改、发布、回滚可还原
5. **扩展性优先**：节点配置、输入输出协议尽量 JSON 化

---

## 三、核心实体总览

```text
Project
  ├── Bot
  │    ├── Workflow
  │    │    ├── WorkflowDraft
  │    │    ├── WorkflowVersion
  │    │    ├── WorkflowNode
  │    │    ├── WorkflowEdge
  │    │    ├── WorkflowVariable
  │    │    ├── SubflowBinding
  │    │    ├── PublishRecord
  │    │    ├── DebugSession
  │    │    └── WorkflowRun / NodeRun
  │    ├── KnowledgeBase
  │    ├── AdvancedKnowledgeBase
  │    └── MaterialAsset
  ├── Role / Permission / ProjectMember
  └── AuditLog
```

## 八、关系说明

### 8.1 一对多关系

- Project -> Bot
- Bot -> Workflow
- Workflow -> WorkflowDraft
- Workflow -> WorkflowVersion
- WorkflowVersion -> WorkflowNode
- WorkflowVersion -> WorkflowEdge
- KnowledgeBase -> KnowledgeDocument -> KnowledgeChunk
- WorkflowRun -> WorkflowNodeRun
- DebugSession -> DebugNodeTrace

### 8.2 关键关联

- `bot.current_workflow_version_id` 指向当前生效版本
- `workflow.current_published_version_id` 指向当前对外生效版本
- `workflow_version.source_draft_id` 保留发布溯源
- `publish_record` 和 `audit_log` 构成治理追踪链路

## 九、索引与分区建议

### 9.1 高频索引字段

- `project_id`
- `workflow_id`
- `workflow_version_id`
- `bot_id`
- `customer_id`
- `created_at / started_at / published_at`

### 9.2 分区建议

建议对以下大表按时间或项目分区：

- `workflow_run`
- `workflow_node_run`
- `debug_session`
- `debug_node_trace`
- `audit_log`

推荐策略：
- 先按月分区
- 数据量进一步增大后，可按 `project_id + month` 做复合分片

## 十、JSON 字段建议内容

### 10.1 definition_json

建议包含：
- 节点列表
- 连线列表
- 画布布局
- 全局变量定义
- 子流程引用关系
- 节点 schema 版本

### 10.2 config_json

建议包含：
- AI 节点提示词、模型配置
- 检索节点的知识库与检索模式
- 输出节点的发送规则
- 渠道节点的目标参数

### 10.3 context_snapshot

建议包含：
- 默认全局变量
- 自定义全局变量值
- 客户属性快照
- 上游节点输出结果
- 调试环境附加信息
