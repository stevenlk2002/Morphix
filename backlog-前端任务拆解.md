# Morphix 前端任务拆解清单

## 1. 文档目标

这份文档是在以下成果基础上继续下钻：

- `design-MVP研发任务拆解清单.md`
- `backlog-后端任务拆解.md`
- `design-接口设计-总控编排与会话运行时.md`
- `openapi-总控编排与会话运行时.yaml`
- `prototype/index.html`

目标是把控制台前端的工作拆成：

- 可以直接分配给前端开发的任务包
- 可以和后端接口逐项对接的页面与模块任务
- 可以按 Sprint 推进的实现顺序
- 可以验收而不是“差不多做了”的交付清单

一句话说：

**这份文档回答的是：Morphix 前端团队明天开工，先搭哪些页面、接哪些接口、做到什么程度算交付。**

---

## 2. 前端范围说明

当前这里说的“前端”，主要指 **控制面 Web 控制台**，不是设备执行端 App。

MVP 阶段前端重点不是把所有原型页面全部工程化，而是优先落地以下控制台能力：

1. 会话列表页
2. 会话详情页
3. 当前运行态展示
4. 人工接管与交还
5. 工作流运行调试页
6. Bot / WorkflowVersion 绑定页
7. 基础鉴权、导航、API 集成、错误处理

也就是说：

**前端 MVP 的本质，不是做一个“全平台后台”，而是先做一个能支撑运营与排障的控制台。**

---

## 3. 已知前端技术基线

结合当前项目记忆与原型，前端建议按以下基线推进：

- **框架**：React 19 + Vite 8
- **UI**：Ant Design 6
- **状态管理**：页面局部状态 + React Query（服务端状态）
- **API 集成方式**：OpenAPI 契约驱动 + Typed fetch 封装
- **实时方案**：MVP 先用轮询查询运行态，不强依赖 WebSocket
- **鉴权方式**：先按控制面 JWT / Session 登录态兼容设计

### 对应架构决策

| 决策项 | 选择 | 原因 |
|---|---|---|
| 前端结构 | feature-first | 页面与业务边界更稳定，便于多人并行 |
| API 客户端 | Typed fetch + React Query | 对当前 OpenAPI 契约最实用，复杂度适中 |
| 服务端状态 | React Query | 适合会话列表、运行态、审计查询这类读多写少场景 |
| 实时更新 | Polling | MVP 成本最低、可回退、便于先打通主链路 |
| 错误处理 | 统一 API Error 映射 + 页面级容错 | 避免各页各写一套异常逻辑 |

## 4. 前端研发原则

## 4.1 先围绕主链路页面建设
优先服务：

`会话查询 -> 进入会话 -> 查看运行态 -> 接管/交还 -> 查看 run / 节点 / 决策 / Agent 调用`

不要先做：
- 大屏报表
- 花哨可视化
- 非核心营销页面
- 复杂多租户权限配置 UI

## 4.2 先 API 驱动，再 UI 丰富化
先把：
- 列表能查
- 详情能开
- 状态能看
- 操作能执行
- 错误能反馈

做出来，再补：
- 高级筛选
- 交互动效
- 批量操作
- 拖拽微交互

## 4.3 先把调试能力内建进去
Morphix 前端不是纯展示台，它还是运行时排障入口。

所以调试页、运行态页、错误态提示，优先级比“装修页面”更高。

---

## 5. 任务优先级定义

- **P0**：不做就无法支撑控制台 MVP 主链路
- **P1**：主链路能跑，但没有会明显影响联调、排障、运营使用
- **P2**：增强项，可在第二轮继续补

---

## 6. 推荐前端模块划分

建议采用 feature-first 结构：

- `src/app/`：应用入口、路由、布局、Provider
- `src/shared/`：API client、错误处理、通用 hooks、UI 基础组件
- `src/features/auth/`：登录态、权限壳、用户上下文
- `src/features/conversations/`：会话列表、详情、消息流水、运行态
- `src/features/handoff/`：人工接管与交还
- `src/features/workflow-runs/`：run 详情、节点轨迹、调试视图
- `src/features/audit/`：策略决策、Agent 调用记录
- `src/features/bots/`：Bot 列表、Bot 与 workflowVersion 绑定
- `src/features/layout/`：导航、顶部栏、页面容器、空态/错误态

---

## 7. 原型到前端页面映射建议

结合 `prototype/index.html`，MVP 前端不需要一次性实现所有原型页，建议先映射以下核心页面：

| 原型意图 | 前端 MVP 页面 | 备注 |
|---|---|---|
| 渠道会话管理 / 三栏会话页 | 会话工作台 | MVP 最核心页面 |
| 会话详情侧栏 / 客户详情 | 会话详情抽屉或右侧面板 | 聚焦运行态与联系信息 |
| 工作流调试节点弹窗 | Run 调试页 / 节点轨迹抽屉 | 给实施/研发排障 |
| 机器人配置 / 工作流绑定 | Bot 配置页 | 先做最小可用版 |
| 顶部导航 / 项目切换 | 应用壳层 | 所有页面共享 |

# 8. P0 任务清单：前端主链路必须项

# 8.1 应用壳层与基础设施

## FE-001 应用入口与路由骨架
- **优先级**：P0
- **Owner**：前端基础
- **目标**：建立控制台应用路由、Layout、页面壳层
- **依赖**：无
- **落点**：
  - `src/app/router.tsx`
  - `src/app/layouts/*`
- **验收标准**：
  - 至少可访问：会话列表、会话详情/工作台、Bot 配置、Run 调试页

## FE-002 全局 Provider 装配
- **优先级**：P0
- **Owner**：前端基础
- **目标**：装配 React Query、主题、通知、错误边界、用户上下文
- **依赖**：FE-001
- **验收标准**：
  - 全局请求缓存可用
  - 全局错误与 Toast 可用

## FE-003 API Client 与 OpenAPI 契约接入
- **优先级**：P0
- **Owner**：前端基础
- **目标**：建立 Typed fetch client，统一 requestId、错误结构、鉴权头附加逻辑
- **依赖**：后端 BE-005
- **接口依据**：`openapi-总控编排与会话运行时.yaml`
- **验收标准**：
  - 前端不直接手写散乱 fetch
  - API 错误可被统一映射

## FE-004 全局错误与空态组件
- **优先级**：P0
- **Owner**：前端基础
- **目标**：统一列表空态、接口错误态、局部加载态
- **依赖**：FE-002、FE-003
- **验收标准**：
  - 会话列表、详情、调试页都有一致的 loading / empty / error 体验

## FE-005 鉴权壳与权限拦截
- **优先级**：P0
- **Owner**：前端基础
- **目标**：建立登录态检测、无权限跳转、过期提示
- **依赖**：FE-002、FE-003
- **验收标准**：
  - 未登录不能进入控制台核心页
  - 401/403 错误有可理解反馈

# 8.2 会话工作台主链路页面

## FE-101 会话列表页 MVP
- **优先级**：P0
- **Owner**：会话模块
- **对应接口**：`GET /api/control/conversations`
- **目标**：实现会话列表、基本筛选、最近消息预览、状态标签
- **依赖**：FE-003、后端 BE-401
- **页面要素**：
  - 项目上下文
  - 渠道账号筛选
  - sessionState 筛选
  - handoffStatus 筛选
  - keyword 搜索
  - 列表项基础摘要
- **验收标准**：
  - 可稳定拉取会话分页数据
  - 点击列表项能进入会话工作台

## FE-102 会话详情工作台布局
- **优先级**：P0
- **Owner**：会话模块
- **目标**：实现会话三栏或二栏主工作台骨架
- **依赖**：FE-101
- **建议布局**：
  - 左：会话列表
  - 中：消息流水 / 聊天区
  - 右：运行态 / 联系人 / 操作区
- **原型参考**：`channel-sessions` 页面布局思路
- **验收标准**：
  - 布局在桌面端可稳定使用
  - 页面切换不丢当前会话上下文

## FE-103 消息流水面板
- **优先级**：P0
- **Owner**：会话模块
- **对应接口**：`GET /api/control/conversations/{conversationId}/messages`
- **依赖**：FE-102、后端 BE-403
- **目标**：展示客户/AI/人工/系统消息流水
- **验收标准**：
  - 支持分页回溯
  - 不同 senderType 有基础区分
  - 加载更多行为稳定

## FE-104 当前运行态面板
- **优先级**：P0
- **Owner**：会话模块
- **对应接口**：`GET /api/control/conversations/{conversationId}/runtime`
- **依赖**：FE-102、后端 BE-404
- **目标**：展示 `sessionState`、`handoffStatus`、`activeRunId`、当前 Bot、当前 WorkflowVersion
- **验收标准**：
  - 运营人员能一眼看出当前会话是谁在控、现在卡在哪

## FE-105 会话详情信息面板
- **优先级**：P0
- **Owner**：会话模块
- **对应接口**：`GET /api/control/conversations/{conversationId}`
- **依赖**：FE-102、后端 BE-402
- **目标**：展示联系人信息、ownerType、最新接管信息、当前 Bot / WorkflowVersion
- **验收标准**：

# 8.3 人工接管主链路

## FE-201 发起人工接管交互
- **优先级**：P0
- **Owner**：接管模块
- **对应接口**：`POST /api/control/conversations/{conversationId}/handoff`
- **依赖**：FE-104、后端 BE-405
- **目标**：实现接管按钮、原因输入、二次确认
- **验收标准**：

- **优先级**：P0

- **依赖**：FE-201、FE-202

- **对应接口**：`GET /api/control/workflow-runs/{runId}/node-executions`

## 13. 前端验收门禁

## 13.1 工程门禁
- 路由结构清晰
- API 调用不散落在页面内到处裸写
- 全局错误处理统一
- 所有关键页都有 loading / empty / error 态

## 13.2 功能门禁
- 能查会话列表
- 能看会话详情与消息流水
- 能看当前运行态
- 能发起人工接管与交还
- 能查看 run / node / decision / invocation
- 能完成 Bot 与 workflow_version 的基础绑定

## 13.3 可用性门禁
- 项目切换后状态不串
- 页面刷新后上下文可恢复
- 接口失败有明确提示
- 慢接口下不白屏

## 14. 前端研发风险提醒

## 风险 1：先做一堆页面装修
后果：看起来很完整，但主链路能力没有站住。

## 风险 2：前端状态管理失控
后果：会话列表、当前会话、run 状态、接管状态彼此打架。

## 风险 3：接口错误处理分散
后果：每个页面一套逻辑，后期维护会很痛。

## 风险 4：把实时刷新做得过重
后果：MVP 阶段复杂度上来，但收益不大。

## 风险 5：没有调试入口
后果：出问题时只能靠后端同学查库。

---

## 15. 最终建议

如果只保留一句最实在的前端建议，那就是：

**先把“运营可观察、可接管、可排障”的控制台做出来，再考虑把它做得更漂亮。**

具体来说，优先级永远是：

1. 列表查得到
2. 会话点得开
3. 运行态看得懂
4. 接管交还做得成
5. run 与节点问题查得到
6. API 异常用户看得明白

只要这 6 件事成立，Morphix 前端 MVP 就不是静态后台壳，而是真能支撑业务运行的控制台。
